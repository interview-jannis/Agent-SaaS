## 1. 스케줄 버전 히스토리 + 리뷰 플로우

### 설계
- `schedules` 테이블이 원래 케이스당 1행(업로드 때마다 in-place update)이었음
- 변경: **케이스당 여러 행 = 여러 버전**. `slug`만 첫 버전에서 공유 (고객 URL 안정)
- 새 컬럼: `file_name`, `revision_note`, `confirmed_at`
- 상태: `pending / confirmed / revision_requested` (CHECK constraint)

### Admin 업로드 2단계
기존: 파일 드롭 → 즉시 DB 커밋 (실수 방지 안 됨)
변경: 스테이징 → 프리뷰(파일명·크기) → `[Confirm Upload]` / `[Cancel]` → 그때 커밋

### Schedule History 섹션
- 모든 버전 카드로 노출, `v2 · filename.pdf · Pending · [View PDF] · [Delete]`
- 최신 버전에 `LATEST` 뱃지
- 상태별 색 (pending blue, confirmed emerald, revision rose)
- Delete 버튼 + window.confirm 경고

### Agent 승인 버튼
Agent Case 상세에 `[Confirm Schedule]` / `[Request Revision]` 추가
- Request Revision → 노트 입력 모달 (필수 입력)
- Confirm → `case.status = schedule_confirmed`

### Admin 잠금
`case.status ∈ {schedule_confirmed, travel_completed}`면:
- 업로드 불가
- Delete 버튼 전부 숨김
- "Agent has confirmed the schedule — no further edits allowed." 배너

### SQL 마이그레이션
```sql
ALTER TABLE schedules ADD COLUMN IF NOT EXISTS file_name TEXT;
ALTER TABLE schedules ADD COLUMN IF NOT EXISTS revision_note TEXT;
ALTER TABLE schedules ADD COLUMN IF NOT EXISTS confirmed_at TIMESTAMPTZ;

ALTER TABLE schedules DROP CONSTRAINT IF EXISTS schedules_status_check;
UPDATE schedules SET status = 'pending' WHERE status NOT IN ('pending','confirmed','revision_requested');
ALTER TABLE schedules ADD CONSTRAINT schedules_status_check
  CHECK (status IN ('pending','confirmed','revision_requested'));

-- slug UNIQUE 제거 (버전 간 공유 허용)
ALTER TABLE schedules DROP CONSTRAINT IF EXISTS schedules_slug_key;
ALTER TABLE schedules ADD CONSTRAINT schedules_case_version_unique UNIQUE (case_id, version);
CREATE INDEX IF NOT EXISTS schedules_slug_idx ON schedules(slug);
```

### Storage RLS
`storage.objects` 테이블은 프로젝트 RLS 껐어도 기본 정책 존재 → 버킷별 `FOR ALL TO anon, authenticated` 정책 추가 필요.

## 3. 고객 정보 확장 — 의료관광 요구사항 반영

### 동기
고객이 준 템플릿(`Travel & Service Overview`)에 맞춰 필드 재정의. 의료관광 특성상 **수술·시술 시 사고 방지** 관점의 정보가 필수.

### 추가된 필드 (공통 필수)
- `emergency_contact_name / relation / phone`
- `blood_type, allergies, current_medications, health_conditions, medical_restrictions`
- `height_cm, weight_kg` (마취 용량, 체성분 검사)
- `preferred_language`, `mobility_limitations`
- `smoking_status`, `alcohol_status` (TEXT + CHECK)
- `pregnancy_status` (female일 때만 노출, male은 자동 `not_applicable`)

### 무슬림 조건부 필수
- `prayer_frequency`, `prayer_location`, `dietary_restriction`
- `same_gender_doctor`, `same_gender_therapist` (Required/Preferred/No preference/N/A)
- `mixed_gender_activities` (Comfortable/Prefer to limit/...)
- `cultural_religious_notes`

### Optional
- `prior_aesthetic_procedures`, `recent_health_checkup_notes`

### Prayer enum 값 재정의
기존 `strict/moderate/flexible` → **`all_five_daily/flexible/not_applicable`** (템플릿 용어에 맞춤)
기존 `prayer_room/mosque_nearby/quiet_private_space/...` → **`hotel/vehicle/external_prayer_room/mosque/not_applicable`**

### 검증 방식
"N/A 허용" — TEXT 필드엔 `N/A` 문자열 입력 허용, ENUM엔 `not_applicable` 값 추가. `trim().length > 0` 통과만 되면 됨.

## 5. Members & Group 관리 — Staging 모드

### 문제 의식
기존 Case 상세의 Members/Companions 섹션은 모든 액션(추가/제거/그룹 배정/Lead 변경)이 즉시 DB 커밋. 잘못 고른 걸 되돌리려면 추가 클릭. 스트레스.

### Staging 방식 도입
`PendingMember` 타입 (isNew, isRemoved, groupId, isLead 등) 배열을 로컬 state로 보관.
- Add existing / Register new → 임시 state에 추가 (isNew=true, 임시 id `new-<clientId>-<ts>`)
- × 클릭 → isRemoved 토글 (↺로 복구 가능, 취소선 + 빨간 테두리)
- Set as Lead → 로컬만 반영
- 그룹 드롭다운 → groupId만 변경
- 변경되면 상단 `Unsaved` 뱃지 + 하단 `[Cancel] [Save Changes]` 노출

### Save 로직 (순서 중요)
1. `toDelete` (isRemoved && !isNew) → case_members DELETE
2. `toInsert` (isNew && !isRemoved) → **upsert** with onConflict `case_id,client_id` (재시도 시 duplicate key 방지)
3. 각 멤버 is_lead 변경분만 UPDATE
4. quote_group_members 재동기화: 그룹 변경분만 delete + insert

### 버그: Save 후 state 안 바뀜
`useEffect`가 `membersDirty` 때문에 서버 데이터로 재빌드 스킵 → 임시 id들이 stale하게 남아있다가 재시도 시 duplicate insert → **fetchCase가 fresh 데이터 반환하게 리팩토링 + Save 성공·실패 모두 명시적 `setPendingMembers(build(fresh))` 호출**.

### 그룹 리더는 안 둠
케이스 전체 리더(Invoice "To:") 하나로 충분. 그룹은 상품 묶음일 뿐.

### 뷰/에딧 모드 분리
Edit 버튼 안 누르면 View 모드 (읽기 전용, 2열 그룹별 compact 리스트). Edit 누르면 전부 편집 가능.

## 6. 정산 흐름 구축

### 흐름 변경
Before: Admin이 `Mark Travel Complete` 누름.
After: **Agent**가 여행 완료 표시 → Admin은 Settlement 페이지에서 확인 → 수동 송금 → SaaS에 기록

### Mark Travel Complete 이동
- Admin Case 상세의 버튼 제거, "Agent will mark travel complete after the trip." 안내만
- Agent Case 상세 Schedule 섹션에 버튼 추가 (schedule_confirmed 상태일 때만)

### Admin Settlement 페이지 (`/admin/settlement`)
- **Unsettled Cases 테이블**: agent · case · lead · 여행일 · **margin · commission** · `[Settle]`
- **Settlement History 테이블**: 같은 컬럼 + paid_at
- Settle 모달: 케이스·에이전트·은행정보 자동 표시, 금액 자동계산(읽기전용), Paid Date만 편집

### Agent Payouts 페이지 (`/agent/payouts`)
- 요약 카드 3개: Unsettled / Received This Month / Total Received
- Unsettled Cases 리스트 + Settlement History
- 요청 기능 없음 (수동 송금 방식)

### DB 변경
```sql
ALTER TABLE settlements ADD COLUMN IF NOT EXISTS case_id UUID REFERENCES cases(id);
CREATE UNIQUE INDEX IF NOT EXISTS settlements_case_id_idx ON settlements(case_id);
```

### 삭제 제외
처음엔 Settlement Delete 버튼 뒀는데 **재무 기록 무결성 원칙**으로 제거. 실수 정정은 Supabase 콘솔에서 직접.

## 10. 공용 유틸 컴포넌트

### DOBPicker
생년월일 native date input이 1970년대까지 스크롤 지옥 → Year/Month/Day 3개 드롭다운. 연도는 현재~100년 역순, 월은 Jan/Feb..., 일은 선택한 월 기준 동적 (Feb 30 → 28/29 자동 보정).

**버그**: 처음엔 `onChange(ny, nm, nd) { if(!all) onChange('') }`로 만들었더니 Year만 골라도 parent에 빈 문자열 커밋 → 컨트롤드라 시각적으로 선택 안 됨. 내부 state로 partial 선택 유지하게 수정.

### PrintPdfButton + AutoPrint
인보이스의 Print 버튼처럼 자동 인쇄 대화창 뜨게 하고 싶었음. PDF는 iframe으로 열어도 `window.print()` 직접 호출이 까다로움 (cross-origin).

**해결**: `/schedule/[slug]?autoprint=1` URL에 새 탭으로 열기 → AutoPrint 컴포넌트가 800ms 후 `window.print()` 호출. iframe 안의 PDF까지 포함해 프린트 대화창 뜸.

### clientCompleteness lib (`src/lib/clientCompleteness.ts`)
여러 페이지(Agent/Admin Case 상세, Agent Clients 리스트/상세)에서 "필수 필드 완비?" 판단 중복. 공용 유틸로 추출:
- `ClientInfo` 타입 (공통 필드 셰이프)
- `getMissingClientFields(c)` → 라벨 배열
- `CLIENT_INFO_COLUMNS` → Supabase SELECT에 재사용

CaseRequiredFields(`concept, meeting_date, outbound_flight, inbound_flight`) 체크도 같이.

## 11. 자잘한 UI 정리

### Date input 텍스트 색
native date input이 브라우저에 따라 회색 텍스트 → `globals.css`에 `input[type="date"/datetime-local"] { color: gray-900 }` 강제. 처음 `:invalid` 예외 추가했더니 값 있는데도 회색 나옴 → 제거하고 무조건 검정.

### Client Info Status 컴팩트화
처음엔 각 미완성 고객마다 풀 카드(LEAD 뱃지 + 17개 필드 나열) → 너무 길다 → **한 줄 요약**: `Client Info Status [2 incomplete] Client 4 · Client 5` (이름만 링크)

### Members Edit 모드도 그룹별 2열
View 모드에서 2열로 그룹 분할했으니 Edit 모드도 동일하게. 각 카드에 이름 · × / 그룹 드롭다운 · Set as Lead.

### Invoice 섹션 토글
Selected Products 전체 숨기기 → 그룹별 소계·전체 총액만 "항상 표시", 상품 세부는 `Show details / Hide details` 토글.

### Admin Overview 통화
Total Revenue / Top Agents 매출을 KRW로 보되 **하단에 작은 회색 글씨로 USD 환산** `≈ $9,185.19`.

### Agent Payouts Summary
Total Earned = Settled + Unsettled로 계산했던 것 → 사용자 지적 "unsettled는 아직 안 받은 건데 Earned라 부르면 안 되지". 카드 재구성: Unsettled / Received This Month / Total Received (실제 입금액 기준).

### Paid Date UI
Amount 입력을 KRW → USD로 변경했다가 "수수료는 이미 결정돼 있는데 입력할 이유가 없지" → **Amount 읽기 전용, Paid Date만 편집** 으로 축소.

## 발생했던 주요 이슈 & 해결

| 이슈 | 원인 | 해결 |
|------|------|------|
| schedules 버킷 not found | Storage 버킷 미생성 | Supabase Storage UI에서 수동 생성 (Public) |
| schedules storage RLS violation | storage.objects RLS 기본 정책 존재 | 버킷별 `FOR ALL TO anon,authenticated` 정책 추가 |
| schedules_status_check 실패 | 기존 'reviewed' 값 | UPDATE → ADD CONSTRAINT 순서 (원샷으로 안 됨) |
| slug UNIQUE 위반 | v2 업로드 시 동일 slug 재사용 | schedules_slug_key DROP + (case_id, version) 복합 UNIQUE |
| Save 후 Unsaved 유지 + duplicate | useEffect가 membersDirty 때문에 rebuild 스킵 | fetchCase 리턴 값으로 명시적 setPendingMembers |
| DOBPicker 선택 안 보임 | 3개 다 안 골랐을 때 onChange('') | 내부 state로 partial 유지 |
| agents 테이블 400 | `created_at` 컬럼 실제로 없음 | order 제거 |
| new client insert 실패 | date_of_birth: '' 빈 문자열 | 필드별 명시 insert + 검증 |
| settlements RLS violation | 테이블 RLS 켜져있음 | ALTER TABLE ... DISABLE ROW LEVEL SECURITY |
| 색 너무 많다 피드백 | 5+4 카드 각자 색 | My Status 전부 gray-50, Action/상태뱃지만 색 유지 |

## 느낀 점 / 메모

- **Staging 모드는 파워 유저에 정말 필요**: Case Members 에서 실수로 잘못 고른 걸 되돌릴 수 있게 되자 UX 체감이 훨씬 좋아짐. 다만 "useEffect dirty 스킵" 버그처럼 refetch 타이밍을 세심히 잡아야 함.
- **date input은 최악**: native date input은 연도 스크롤, 색상, 포맷 모두 브라우저 제멋대로. 중요 UX엔 커스텀 컴포넌트(DOBPicker 같은) 만드는 게 백배 낫다.
- **RLS는 조용히 켜짐 주의**: 이번에도 settlements · agents 테이블 RLS가 어느 시점에 켜져있었음. 테이블 추가할 때마다 즉시 끄는 걸 루틴화해야.
- **파일 커밋 빈도**: 오늘 3000+ line 변경 생겼는데 중간에 커밋 안 한 구간 길었음. 기능 단위로 커밋하는 습관 회복 필요.
- **정보량과 가독성의 갈등**: Dashboard 3번 리디자인한 것 교훈. "다 보이는 게 친절" 아니고, "한눈에 읽히는 게 친절". Status 5개 카드 컬러풀 → 전부 회색 → 0이면 죽이고 1+면 컬러. 점진적 개선이 정답.
- **정산 금액 입력은 불필요**: 수수료 계산식이 deterministic이므로 admin이 금액 입력할 필요 없음. 읽기 전용 + Paid Date만 받는 걸로 대체. "입력할 필드가 의미 있는가?"를 항상 물어야.
