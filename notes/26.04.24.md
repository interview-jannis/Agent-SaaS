## 1. 초대 링크 온보딩 — Temp 계정 대체

### 기존 문제
- Admin이 `temp01/temp01` 같은 credentials 핸드오프 → 보안 취약 (비번 예측 가능)
- `agents` 테이블 지워도 `auth.users`는 남아서 **번호 충돌 회피 로직** 필요
- Temp 계정 청소 부담

### 새 구조
```
[Admin] "Invite Agent" 버튼 → 1회용 토큰 생성 + placeholder auth user
  → 모달에 `/invite/{token}` URL + Copy 버튼
  → Admin이 WhatsApp/메신저로 URL 전달

[Agent] 링크 클릭 → /invite/[token]
  → POST /api/onboarding/claim (토큰 검증 → placeholder email/password 반환)
  → supabase.auth.signInWithPassword → /onboarding으로 redirect

[흐름] OT → NDA(이름/국적) → Partnership → awaiting_approval
  → Admin 승인 → Setup Wizard (실제 email/password 입력) → Agent Home
  → Setup 완료 시 invite_token/invite_secret NULL 처리 (1회용)
```

### DB 추가
```sql
ALTER TABLE agents ADD COLUMN invite_token TEXT UNIQUE;
ALTER TABLE agents ADD COLUMN invite_secret TEXT;  -- placeholder password, cleared on setup
ALTER TABLE agents ADD COLUMN invited_at TIMESTAMPTZ;
ALTER TABLE agents ADD COLUMN invite_expires_at TIMESTAMPTZ;  -- +7일 만료
```

### 결정
- `/invite/[token]`을 **루트 경로**에 위치 (`/onboarding/invite/...` 아님). Onboarding layout이 auth 세션 필요하기 때문 — 토큰 클릭 시점엔 아직 미로그인
- placeholder email: `invite-{token12}@tiktak.temp` (기존 `@tiktak.temp` suffix 유지해서 Setup Wizard의 기존 검증 로직 재사용)
- Admin Agent 상세에 **"Invite Link 섹션"** 추가 → 링크 복사 놓쳐도 재확인 가능
- 기존 `/api/admin/create-agent` 제거, `/api/admin/invite-agent` 신규

### 교훈
- 온보딩 라우팅 가드 **3곳 동기화 필수**: Login / AgentOnboardingGuard / Onboarding layout — 한 군데 놓치면 리다이렉트 루프 발생
- 초대 방식은 보안/UX 모든 면에서 우위. "이메일 모름 케이스" 우려도 token-only 방식으로 동일하게 커버 가능

## 2. Agent Reject (soft) + Delete (hard) 분리

### Reject — 재도전 가능
```sql
ALTER TABLE agents ADD COLUMN rejection_reason TEXT;
ALTER TABLE agents ADD COLUMN rejected_at TIMESTAMPTZ;
```

- Admin이 사유 입력 → `rejection_reason`/`rejected_at` 저장 + 계약서 삭제 + status `pending_onboarding`
- Agent Waiting 페이지가 **15초 폴링 중 rejection 감지** → "Your application needs another look" 화면 전환
- Start Over 버튼 → rejection 클리어 + `/onboarding/nda`로 이동
- ContractStep이 Partnership 재서명 시에도 rejection 자동 클리어 (방어)
- Approve 시에도 클리어 (미래 재사용 방지)
- Admin 리스트 "Rejected" rose 뱃지로 구분

### Delete — 완전 제거
- `/api/admin/delete-agent` 신규 — agent_contracts → clients → agents → auth.users 순서
- **Case 1건이라도 있으면 409 차단** ("Deactivate instead" 메시지)
- Agent 상세 "Danger Zone" 섹션 (approved 아닌 agent에만 노출)

### 교훈
- 기존 Reject 구현이 절반뿐이었음: contracts만 삭제하고 Agent에게 사유 전달 경로 없음 → "왜 서명했는데 아무 반응 없나" 혼란
- Waiting 페이지의 polling 로직이 승인만 감지했음 → rejection_reason도 감지하도록 확장

## 3. Invoice/Schedule "고객 열람" 알림

### 접근
- `/quote/[slug]`, `/schedule/[slug]` 서버 컴포넌트에서 요청 시점에 직접 처리
- 첫 열람: `first_opened_at` 기록 + `notifyAgent` → Realtime bell 알림
- 재열람: `open_count`만 증가 (중복 알림 방지)
- Agent/Admin의 내부 preview 링크엔 `?preview=1` 파라미터 → 기록 스킵
- Schedule은 `?autoprint=1` (기존 Agent print용) 도 함께 스킵

### DB 추가
```sql
ALTER TABLE schedules ADD COLUMN first_opened_at TIMESTAMPTZ;
ALTER TABLE schedules ADD COLUMN open_count INTEGER DEFAULT 0;
-- quotes에는 이미 있었음 (first_opened_at, open_count)
```

### Admin 알림 안정화
- Agent 계약 완료 시 알림이 종종 안 가는 문제 발생
- 원인 불명 (client-side `notifyAllAdmins`가 silent fail 가능)
- **`/api/onboarding/notify-signed` 서버 API로 교체** (service role 사용) — admin broadcast를 서버에서 insert
- ContractStep이 이 API 먼저 호출, 실패 시 client-side broadcast로 fallback

## 4. Audit Log 타임라인 개편

### 이전
- 행마다 액션 뱃지(pill) + actor → target + details + timeago
- 모든 행이 동등 무게 → 스캔 어려움

### 변경
- **날짜 그룹 헤더** (Today / Yesterday / Mon, Oct 22) + sticky
- 각 행: `[카테고리 아이콘 24×24 tinted] [시간 14w 고정폭] [문장]`
  - 문장: "Admin Nari approved #AG-003 Ahmed" (볼드 actor + 회색 verb + 모노 target)
- ACTION_TONE 매핑: positive(emerald) / negative(rose) / info(blue) / schedule(violet) / pending(amber)
- 세부 details 중 `amount/_krw` 키는 천단위 콤마, `reason` 키는 따옴표로 감쌈
- `agent.deleted` 액션 추가 (기존에 빠져있어서 raw text로 노출됐던 버그 수정)

### 레이아웃 듀얼 모드
**작업 모드 (좌측정렬 + `px-12`)**: Overview, Cases, Products, Agents, Clients, Settlement, Audit Log
**설정 모드 (중앙 + `max-w-xxx`)**: Contracts, Settings, Profile

이전엔 모두 `mx-auto` 중앙 정렬 → 와이드 모니터에서 "왼쪽이 비어있는 느낌" → Admin Settlement / Audit Log / Agent Payouts / Dashboard 4개를 좌측정렬로 전환.

## 5. Agent Payouts 재설계

### 여정 (반복)
1. 초안: Hero + 4 KPI 카드 + 큰 월별 차트 + 2-col (Unsettled + Bank) + Settlement History
2. 피드백: "너무 많고 정리 안 돼" → **주인공 발굴 필요**
3. 합의: **Settlement History가 주인공** (Agent가 페이지 여는 첫 이유 = "받았나?")
4. 재구성: Hero → Settlement History → Waiting → Bank
5. 추가: 월별 차트 요청 → 금액 바 + 환자 수 선 **콤보 차트**
6. 반복 수정: 바 폭 / 정렬 / Y축 / 호버 툴팁 / nice number 스케일링

### 최종 구조
```
[HERO] Total Received (큰 숫자) + This Month · 6개월 sparkline (선택)
[Monthly Performance 차트] 금액 바 + 환자 수 선 콤보 (dual Y-axis, hover 툴팁)
[Settlement History — 주인공, Paid On 기준 정렬]
[Waiting to be Paid — amber 톤 제거, 중립 tone]
[하단 2열: Monthly KPI / Bank Information 이동]
```

### 결정
- **Bank Info를 Profile → Payouts로 이동**. Settlement 관련 정보를 한곳에 모음
- **Sparkline** 초반엔 Hero 내부에 있었으나 데이터 1개월뿐일 때 허전 → 아래 큰 차트로 통합
- **nice number ceiling**: maxValue가 $12,686이면 축은 $15K/$20K로 올림. `1/2/5 × 10ⁿ`
- **차트 SVG**: 라인은 SVG polyline (viewBox stretching), **점은 absolute div** — SVG circle은 `preserveAspectRatio="none"`에서 타원으로 짜부라지는 버그 회피
- **바 vs 선**: "막대로 하면 좋을까 선으로 하면 좋을까" 질문에 → **콤보가 정답** (월별 덩어리는 바, 환자 수 추세는 선)

## 6. Agent Dashboard 재설계

### 반복
- 초안: Hero + Action Needed + 2×2 (Recent Cases + Upcoming Travel) + Pipeline 5칸
- "중구난방" 피드백
- 논의 계속되며 **정보가 중복되는 구조** 발견:
  - Pipeline (현재 상태)
  - Recent Cases (최근 생성)
  - Upcoming Travel (미래 여행)
  - 이 셋이 **같은 케이스의 다른 슬라이스** → 사용자 입장 중복

### 해법 — 기능을 통합한 Kanban Pipeline
각 상태 셀을 **Kanban 컬럼**으로:
- 헤더: STATUS 라벨 + 카운트 (tinted)
- 본문 (white): case#+name + **status-specific 맥락 라인**
  - Awaiting Payment: "Due Apr 27 · $18.3K" (overdue면 red, 임박하면 amber)
  - Payment Confirmed/Schedule 단계: "Travel Apr 28" (7일 이내면 amber)
  - Travel Done · Unpaid: "Ended Apr 24 · $1.1K" (기다리는 commission)
  - Settlement Paid: "Paid Apr 29 · $12.7K" (emerald)

→ Action Needed, Upcoming Travel, Recent Activity **전부 삭제 가능**. Pipeline 하나가 "상태 + urgency + 맥락"을 다 커버.

### 최종 3섹션
```
[Header] Quick Actions: + Add Client / + Create Quote
[Hero] This Month (큰 숫자) · Expected Pipeline (우측) · 전체 폭 Tier bar
[Pipeline] 6 Kanban 컬럼 (1열 × 6 상태)
```

### 결정
- **Travel Completed 카운트 분리**: 정산된 건 Settlement Paid 컬럼으로. 이전엔 "travel_completed=3" 하나에 모여서 실제 3건 진행처럼 보였음
- `PIPELINE_LABELS` 별도 정의 — Pipeline에서만 "Travel Done · Unpaid"로 표시 (Recent Cases 뱃지는 그대로 "Travel Completed")
- Hero는 **5xl 큰 숫자 + 좌우 정렬** (text-5xl, text-6xl은 과함), 하단 풀폭 progress bar 분리

### 교훈
- "뭐가 더 필요할까"보다 **"뭐를 빼도 될까"** 가 좋은 질문이었음
- 케이스 정보를 Pipeline에 넣으면 다른 섹션과 중복 → **Pipeline 하나로 통합**하는 게 깔끔
- 데이터 희소할 때 "모든 기능이 빈 상태로 보이는" UX가 훨씬 약점

## 7. Admin Overview 재설계

### Admin 관점 분석
1. "내가 뭐 해야 하지?" (액션)
2. "사업 어떻게 돼?" (재무)
3. "병목 있어?" (정체)
4. "누가 대기 중?" (승인)

### Hero — 4지표로 완성
기존 "Monthly Revenue" 하나 → **Revenue / Earnings / Partner Costs / Agent Payouts**

```
Revenue ($) ₩ from N paid cases · M new clients  |  Earnings ($) ₩ company margin only
Partner Costs ($) ₩ to hospitals/partners         |  Agent Payouts ($) ₩ settlements sent
```

### 계산식 (중요)
```
total_price = base × (1 + co_margin) × (1 + agent_margin)
base        = total_price / ((1+co) × (1+agent))
Revenue     = Σ total_price          (고객이 낸 돈)
Partner Cost= Σ base                 (병원에 지급)
Earnings    = Σ base × co_margin     (회사 실제 마진)
Agent Payout= Σ settlements.amount   (에이전트 송금분)
```

검산: `base + base·co + base·(1+co)·agent = base · (1+co) · (1+agent) = total_price ✓`

### 버려진 "Net" 라벨
초기엔 `Net = Money In − Agent Out` 제시했다가 **오해 소지 큼** 지적:
- Money In은 **gross** (base 포함) → Admin이 Net을 "이익"으로 오해
- 수정: Net 제거, Earnings를 별도 지표로 (회사 실질 수익)

### Action Required 통합 큐
5가지 조건을 urgency 색으로 구분해 한 섹션에:
- 🟣 Pending agent approvals (신규)
- 🔴 Payments overdue (신규)
- 🟡 Payment to confirm
- 🔵 Schedule upload needed
- 🟠 Stuck cases 5+ days (신규)

Recent Cases는 제거 (Cases 탭 중복).

## 8. Admin Settlement 재설계

### Admin 업무 플로우 재현
- 실제 송금 단위는 **Agent별** (한 번 송금 = 한 명에게 여러 케이스 합산)
- 현재 구조는 case 단위 리스트 → **case 하나씩 Settle 버튼 누르는 게 비효율**

### Agent별 그룹핑
```
[Hero] Paid This Month / Pending Payouts (amber) / Total All-Time

[Pending Settlements] grouped by agent · oldest waiting first
  ▼ Ahmed Al-Rashid #AG-001  $1,125  (1 case · 14d waiting)
     · #C-001 Client 1 · 2026-04-24 · 15% · $1,125  [Settle]

[Settlement History] Paid On 기준 재정렬
```

### 정렬
- 가장 오래 기다린 에이전트 상단 (공평성)
- 14일 이상 대기 → red badge

### Bank info 누락 경고
초기엔 그룹 헤더에 경고 뱃지 추가했으나, **Setup Wizard에서 Bank 필수 입력**이라 실제로 누락될 일 없음 → 경고는 유지만 (defensive), 문구는 제거하지 않음

## 느낀 점 / 메모

- **"뭐 넣을까"보다 "뭐 뺄까"**: 오늘 재설계 대부분이 기능 추가가 아니라 **정리/삭제**였음. Recent Activity, Upcoming Travel, Action Needed 섹션 모두 Pipeline이 맥락 충분히 담으면 사라져도 됨. 정보 밀도보다 **정보의 역할 분리**가 핵심.
- **"Agent라면 어떨까"의 반복 효과**: 내가 제안한 초기 구성은 "예쁘게 쌓기"였지만, 매번 "Agent 관점에서 이게 정말 필요해?" 질문에 걸러지면서 더 적어지고 더 쓸모 있어짐. 같은 원칙을 Admin Overview/Settlement에도 적용했고 훨씬 실무적 결과.
- **Net 라벨 실수의 교훈**: 숫자 자체가 틀린 건 아닌데 **라벨이 암시하는 의미**가 잘못이면 UI로 거짓말하는 셈. Admin이 "이번 달 이익 $181K"로 오인하면 사업 판단 잘못할 수 있음. 재무 지표는 **용어 정확성** 절대적으로 중요.
- **바-선 콤보 차트**: 두 지표(금액 vs 환자수)를 한 차트에 넣을 때 **두 메트릭 성격이 다르면 다른 형태로 그리는 게 최선**. 금액은 덩어리(바), 환자수는 추세(선). 같은 모양 두 개는 헷갈림.
- **SVG circle vs CSS div**: SVG에 `preserveAspectRatio="none"` 쓰면 circle이 타원으로 찌부러짐. 해결은 CSS div로 점 렌더링. 저지르고 나서 명확해짐 — 첫 시도 때 testing 하지 않은 편향이 있었음.
- **Invite Link 재확인 UX**: 초기 구현에서 모달 Done 누르면 링크 다시 못 보는 버그 있었음. Admin Agent 상세 페이지에 **항상 노출**하는 섹션 추가로 해결. 1회성 UX의 함정 — 복사 실패 시 회복 경로 필요.
- **초대 플로우 variant A vs B**: "placeholder 계정을 미리 만들지 말고 서명 끝나고 만들자"(A) vs "placeholder 계정 먼저 만들고 Setup에서 덮어쓰자"(B). A가 구조적으로 깔끔하지만 AgentOnboardingGuard/Waiting 폴링/Login 분기가 auth 세션 전제로 짜여있어서 **A로 가면 전부 재작성**. B는 기존 로직 그대로 재활용 가능. 일정상 B가 정답.
