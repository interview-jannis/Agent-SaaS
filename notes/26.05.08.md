## 2. Commission Invoice Mark Paid 통합

기존에 "Commission Invoice" 카드와 별개로 "Agent Settlement" 섹션이 있었는데,
흐름이 중복이고 UX가 복잡함. Commission Invoice 카드에 Mark Paid를 직접 달고
"Agent Settlement" 섹션 전체 제거.

### 변경 사항 (`src/components/CaseDocumentsSection.tsx`)

**Mark Paid 핸들러 (`doMarkPaid`)**:
commission_invoice 타입이면 결제 확인 후:
1. `settlements` 테이블에 신규 row 삽입 (settlement_number `#S-XXX` 자동 발번)
2. `cases.status` → `completed` (현재 `awaiting_settlement`인 경우에만)
3. agent에게 커미션 지급 알림 전송

```typescript
} else if (doc.type === 'commission_invoice') {
  const amount = doc.total_price ?? 0
  const { count } = await supabase.from('settlements')
    .select('*', { count: 'exact', head: true })
  const next = (count ?? 0) + 1
  const settlementNumber = `#S-${String(next).padStart(3, '0')}`
  await supabase.from('settlements').insert({
    settlement_number: settlementNumber,
    agent_id: agentId,
    case_id: caseId,
    amount,
    paid_at: paidAtValue,
  })
  await supabase.from('cases')
    .update({ status: 'completed' })
    .eq('id', caseId)
    .eq('status', 'awaiting_settlement')
  if (agentId)
    await notifyAgent(agentId,
      `${caseNumber} Commission paid (${docNumber}) — ${fmtUSD(amount / exchangeRate)}`,
      `/agent/cases/${caseId}`)
}
```

**`canEdit` 분기**: commission_invoice는 `actor === 'admin'` (발행자가 agent여도 admin만 결제 확인).
기존: `actor === doc.from_party`.

**`+ Add item` 제거**: commission_invoice와 deposit_invoice는 Add item 버튼 비노출.

**커미션 모달 금액**: KRW+USD → USD만. `return fmtUSD(krw / exchangeRate)`.

**Agent Settlement 섹션 전체 제거** (`src/app/admin/cases/[id]/page.tsx`):
- state: `agentSettlePaidAt`, `savingAgentSettle`, `agentSettleError` 제거
- JSX: Agent Settlement 섹션 전체 (~70줄) 제거

## 3. Pre-confirmation 상품 노출 정합성

### 문제

원래 설계: schedule 확정 전에는 원본 항목(original)만 보임. admin이 추가하거나 삭제한
변경사항은 확정 전까지 "미확정". 그러나 `removed_at`으로 soft delete된 원본 항목이
agent 쪽에서 사라져 버리는 문제.

원칙 재확인:
- 확정 전 삭제는 "삭제 요청" 상태 — 아직 확정 아님
- agent는 원본 항목 전체를 계속 봐야 함
- admin_added 항목만 확정 전 미노출

### 수정 (`src/components/SelectedProductsSection.tsx`)

```typescript
function activeItems(g: Group, showOriginalOnly = false): Item[] {
  if (showOriginalOnly) {
    // 확정 전: removed_at 무시하고 original 항목 전체 노출
    return (g.document_items ?? []).filter(it =>
      it.origin === 'original' || it.origin == null
    )
  }
  return (g.document_items ?? []).filter(it => !it.removed_at)
}
```

### 수정 (`src/app/agent/cases/[id]/page.tsx`)

`totalKrw` 계산 — pre-confirmation: removed_at 무시하고 original 항목 합산:


## 6. Notes from Agent / Notes for Tiktak 스타일 분기

### Admin 쪽 (Notes from Agent)

케이스가 완료 단계(`awaiting_review / awaiting_settlement / completed`)에 들어가면
빨간 테두리 강조 제거 — admin이 읽어야 할 시급성이 없어진 시점이므로.

```typescript
const isDone = caseData.status === 'awaiting_review'
  || caseData.status === 'awaiting_settlement'
  || caseData.status === 'completed'
<div className={`rounded-xl p-3 space-y-1 ${
  isDone ? 'bg-gray-50 border border-gray-200' : 'bg-white border-2 border-red-300'
}`}>
```

### Agent 쪽 (Notes for Tiktak)

Agent가 직접 쓴 메모라 강조 유지가 필요한 이유는 다르다:
- 평소(active 단계): 빨간 테두리 유지 — "admin에게 전달해야 할 중요 정보임"을 상기
- `completed`일 때만 gray mute — 케이스가 완전히 끝난 뒤에는 불필요

`awaiting_review / awaiting_settlement`는 여행 후 진행 중인 단계라 agent가 메모를 참조할 수 있으므로 강조 유지.

```tsx
<section className={`rounded-2xl p-5 space-y-3 ${
  caseData.status === 'completed'
    ? 'bg-gray-50 border border-gray-200'
    : 'bg-white border-2 border-red-300'
}`}>
  <h3 className={`text-xs font-semibold uppercase tracking-wide ${
    caseData.status === 'completed' ? 'text-gray-500' : 'text-red-600'
  }`}>Notes for Tiktak</h3>
```

## 8. Additional Invoice — variant 선택 + 가격 자동입력

추가 인보이스 Add item 폼에서 상품 선택 시 가격이 비어있어 매번 수동 입력해야 했음.

### 변경 사항 (`src/components/CaseDocumentsSection.tsx`)

- `Product` 타입에 `product_variants` 배열 추가, fetch 쿼리에 join
- `autofillPrice(productId, variantId, setter)` 헬퍼:
  - variant 1개: 즉시 KRW 가격 채움 (USD이면 exchangeRate 환산)
  - variant 2개+: variant 선택 후 채움
  - variant 없음: `base_price` 그대로
- **Variant dropdown**: 상품 선택 후 variant가 2개 이상이면 드롭다운 등장
- 상태 2개 추가: `newItemVariantId` (카드 인라인), `draftVariantId` (issue 모달)
- `AdditionalItemDraft` 타입에 `variantId`, `variantLabelSnapshot` 추가 (`src/lib/documents.ts`)
- `issueAdditionalInvoice` → `addDocumentItem` 호출 시 variant 필드 전달

### `+ Additional` 버튼 `completed` 시 숨김

```tsx
{actor === 'admin' && caseStatus !== 'completed' && (
  <button …>+ Additional</button>
)}
```

여행 끝나고 케이스가 완료된 뒤에는 추가 청구가 없으므로 버튼 제거. Partner Payouts는
병원/호텔 정산서가 늦게 오는 경우 있어 그대로 유지.

## 정성적 메모

- `awaiting_settlement` 단계: 여행 후 리뷰 단계와 정산 단계를 분리해서 관리 흐름이
  명확해짐. commission invoice Mark Paid 이후에야 케이스가 `completed`로 넘어가므로
  정산 누락 방지.
- settlement 생성을 commission_invoice Mark Paid 핸들러에 직접 박는 방식이 Admin Settlement
  섹션 따로 두는 것보다 훨씬 단순함. 케이스 컨텍스트 안에서 끝남.
- Pre-confirmation origin 필터: removed_at 무시 + original만 — 논리가 단순하고
  agent/quote/financials 세 뷰 모두 일관됨. 확정 전 삭제는 "삭제 제안"이지 확정 삭제가 아님.
- Client Review 섹션: admin이 case 페이지에서 바로 설문 응답 확인 가능.
  `/admin/surveys` Responses 탭과 중복이지만 case 컨텍스트에서 바로 보이는 게 운영에 편리.
