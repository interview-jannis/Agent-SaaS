## 1. 편집 잠금 정책 정착

### canceled 케이스 — view-only
- `cases.cancellation_reason` 컬럼을 fetch select에 추가, agent/admin Case 상세 양쪽에 read-only 배너
- 배너 문구: "This case has been canceled / Cancellation reason: ... / Editing is disabled. View-only."
- 처음에 "Editing is disabled" 한 줄에 reason 붙였더니 "편집 불가의 이유가 ~~"처럼 읽혀서 분리 — `Cancellation reason:` 라벨 별도 줄.
- Agent 쪽: Travel Dates / Trip Info / Members / Send 버튼 모두 hide. save 핸들러에 early return 가드.
- Admin 쪽: 액션 대부분 status 양성 매칭(`awaiting_*`)으로 이미 자연 차단. 배너만 추가.

### Client 필수 필드 clear 방지
- 한 번 채워진 필수 필드는 빈값으로 저장 시도 시 차단. 값 변경은 OK.
- Muslim 토글 off는 자동 정리이므로 허용 (Muslim-only 필드만 일괄 클리어).
- 에러 메시지: `"Cannot clear required fields once set: Passport, Allergies. Update the value instead."`

### Trip Info / Members — schedule_confirmed 이후 잠금
- 잠금 상태: `awaiting_pricing / awaiting_payment / awaiting_travel / completed / canceled`
- `tripMembersLocked` 플래그 추가. Travel Dates / Trip Info / Members Edit 버튼 모두 hide.
- save 핸들러에도 early return.
- 미팅 결정 명분: "schedule 컨펌 = admin이 그 정보 기반으로 작업 진행한 시점. 그 후에 변경되면 무결성 깨짐."

### Trip Info / Members — `awaiting_schedule` 단계에선 수정 허용 + clear 차단
- 흐름 B 적용 후 status가 `awaiting_schedule`로 자동 승격된 케이스에서, Trip Info를 수정하다가 빈값 저장 → admin 작업 중인 정보 사라짐 위험.
- `caseData.status !== 'awaiting_info'`면 빈값 저장 차단(필수 필드만), 값 변경은 OK.
- Members도 동일 — 그룹 슬롯이 한 번 채워졌으면 미달 상태로 저장 차단. "Replace the missing member instead."

### isComplete 그룹 슬롯 검증 추가
- 자동 승격 버그 수정: `caseTransitions.ts`의 `isComplete()`가 그룹 배정 완성도를 안 보고 trip info + member 정보만 보고 OK 판정 → 비행 날짜만 채우고 그룹 미배정인 상태에서 `awaiting_schedule`로 자동 승격됐었음.
- 모든 quote_group의 `quote_group_members.length === member_count`여야 통과. quote 없거나 그룹 0개면 미완성.

## 2. /quote vs /invoice 라우트 분리

### 문제
이전 구현: 단일 `/quote/[slug]` 라우트가 `quote.finalized_at` 기준으로 Quotation/Invoice 분기 + `?as=quotation|invoice` 파라미터로 강제 모드.
- URL이 둘 다 `/quote/`로 시작 — 의미상 혼란.
- Admin preview 링크가 `?as=`로 갈라져 보기 어려움.

### 해결
- 신규 `src/components/QuoteDocument.tsx` — 공용 server component (slug/mode/preview prop)
- `/quote/[slug]/page.tsx` — 항상 Quotation 모드
- 신규 `/invoice/[slug]/page.tsx` — 항상 Invoice 모드
- `?as=` 파라미터 제거 (라우트가 곧 mode)
- Send 버튼이 `quote.finalized_at` 기준으로 `/quote/` 또는 `/invoice/` 자동 선택
- Admin Case 상세 preview 링크 2개(`Quotation ↗` / `Invoice ↗`)는 각각 다른 라우트로
- 알림 트래킹은 mode-aware: invoice 라우트만 invoice 열람 카운트, quote 라우트는 quotation 열람만

## 3. 알림 메시지 정밀화 (양방향)

### Agent → Admin 변경 알림에 diff 명세
- `notifyCaseInfoChanged(caseId, change?: { header, items })` API로 통일.
- 호출 측은 항목 배열만 만들고 포맷은 lib에서 일관되게:
```
#C-XX Trip info updated

• Concept: K-Beauty + Medical → K-Beauty + Medical + Wellness
• Outbound departure: 2026-05-18 22:00 → 2026-05-19 00:30
• Outbound departure airport: DXB → AUH
```
- Trip Info: concept + 비행 4 sub-field 각각 old → new
- Members: added/removed names, lead 전환 (old → new), 그룹 reassignment per member, 그룹 rename per group
- Client info: 변경된 필드명만 (의료/특이사항 텍스트는 너무 길어져 old → new는 생략)

### Admin → Agent 변경 알림
- **Schedule 재업로드 메모 (manual)** — `schedules.admin_note` 컬럼 추가. Admin upload UI에 "What changed?" textarea. v1은 optional, v2+는 placeholder가 revision 응답 예시 강조.
  - 알림: `"#C-XX Schedule v2 uploaded — Moved hospital to afternoon, swapped day 3/4"`
  - Admin/Agent Schedule History 박스에 blue "Admin Note" 박스로 노출
- **Pricing reprice 자동 diff** — first finalize는 `"Pricing finalized — invoice ready to send"` 그대로. Reprice일 때만:
  - Total 변경: `"Total ₩99,765,325 → ₩100,765,325"`
  - Item 변경: `"2 items repriced"`
  - Due date 변경: `"Due 2026-04-29 → 2026-04-28"`
- Total 변경 있을 때만 `"Please review before resending."` 문구 자동 추가.

### 단일 변경도 bullet 형태로 통일
처음엔 변경 1개면 한 줄("— Total ..."), 2개+면 bullet으로 가르려 했는데 사용자 피드백: "닷 찍히는 게 더 깔끔". 통일.

### NotificationBell whitespace-pre-line
`<p>` 메시지에 `whitespace-pre-line` 추가 — `\n` 그대로 줄바꿈으로 렌더.

### 알림 클릭 시 새로고침
- 다른 페이지면 router.push (기존)
- 같은 페이지면 `window.location.reload()` (case/client 상세는 client-side fetch라 router.refresh()로는 useEffect 재실행 안 됨)

### Admin 알림 broadcast 신뢰성 회복
- 증상: agent의 Mark Travel Complete / Schedule Confirm / Revision Request 등에서 admin에게 알림이 안 옴.
- 원인: `notifyAllAdmins`가 client-side INSERT로 admin의 auth_user_id 행을 만들 때 RLS/세션 경계 문제로 조용히 실패. 4/24 노트에 "server API(service role) 우선 + client fallback" 결정이 있었는데 onboarding 시그니처에만 적용됐고 다른 broadcast는 client-only였음.
- 수정:
  - 신규 `/api/notifications/broadcast-admins` (service role, generic)
  - `notifyAllAdmins`가 브라우저면 서버 엔드포인트 먼저 호출, 실패 시 client fallback
  - 호출 측 코드 변경 없음

## 5. 기타 UX 폴리시

### Quotation/Invoice 그룹 라벨
- 처음엔 "Parent" 같은 그룹명만 표시. 그룹 3개+면 헷갈림.
- "Group 1: Parent", "Group 2: Smith Family", "Group 3: ..." 형태로 인덱스 prefix 추가.

### Admin Schedule History
- "View PDF ↗" → eye 아이콘 + "Preview" (agent 통일)
- 처음엔 섹션 헤더에 latest preview 1개만 노출 → 사용자 피드백: "admin은 버전 마다 다 볼 수 있어야" → 각 박스 상단 우측에 Preview + 날짜 같이.
- `/schedule/[slug]`이 항상 latest 보여주는 버그 발견 → `?v=` 파라미터 추가, admin preview 링크는 `&v=${s.version}` 명시.
- Delete 게이트 강화: 기존 `pending && isLatest`에 `&& !first_opened_at` 추가 — 클라이언트가 한 번이라도 열었으면 삭제 차단.
- Delete 시 agent 알림 발송 (`"#C-XX Schedule v1 deleted by admin"`).

### Financials 박스 — awaiting_pricing 안내 배너
- 스케줄 컨펌 → 인보이스 발급 대기 상태에서 agent가 뭐 해야 하는지 모를 수 있음.
- Blue banner: "Final invoice in preparation — Schedule has been confirmed. Admin is finalizing the pricing — the official invoice will be ready shortly. You'll be notified once it's available to send to your client."
- Send 버튼은 `awaiting_pricing` 동안 hide (인보이스 없으니 무의미).

### Mark Travel Complete — 그냥 버튼
- 날짜 지정 vs 버튼만 — 쇼핑몰 구매확정 패턴으로 결정. 버튼만.
- 이유: travel_end_date(계획)가 정확한 종료일 보존, travel_completed_at은 "agent 컨펌 시점"으로 의미가 명확. 정산은 당월 환자수 기준이라 며칠 차이 무관. UX 단순.

## 느낀 점

- **알림 broadcast 침묵 실패가 의외로 컸다**. 4/24에 onboarding만 server API로 우회하고 다른 건 client 그대로 둔 게 누적 부채. 시뮬레이션 들어갔으면 "admin이 알림 못 받는다"가 반복 신고됐을 것. 한 번에 generic 엔드포인트로 정리.
- **알림 메시지 diff 명세는 흐름 B의 가치 증폭기**. admin이 "뭐가 바뀌었지?"로 케이스 들어가는 횟수 자체를 줄임. agent도 admin이 가격 어떻게 조정했는지 알림에서 바로 확인.
- **"하나 업데이트 되는 것도 닷 찍히게"** — 작은 일관성 요청이지만 의미 있음. 1개일 때만 한 줄 처리하면 형식이 두 가지가 되고, 사용자가 "어떤 형식으로 올지" 예측 못 함. 통일이 곧 신뢰.
- **편집 잠금 3단계** (canceled / schedule_confirmed / client 필수 필드) 정착 — 이제 어떤 단계에서 무엇이 read-only인지 명확. agent가 멘붕할 시나리오 대부분 차단.
- **admin_note (재업로드 메모) ↔ revision_note (revision 요청)**: 양방향 대칭 구조 완성. 둘 다 PDF 자체로는 자동 diff 불가능한 영역의 manual 보완.
