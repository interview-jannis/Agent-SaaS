## §3. Partner Payouts 이중 곱셈 버그 수정

**파일**: `src/app/admin/cases/[id]/page.tsx`

### 문제
`document_items.base_price` = `baseUnitKrw × multiplier` (quotation 생성 시 총액으로 bake-in).  
기존 코드: `itemCost = base_price × item.quantity` → multiplier가 두 번 곱해져 ₩882M 오계산 (실제 ₩365M 수준 대비).

### 수정된 공식
```
base_price          → 일반 아이템 총비용 (이미 단가×수량 bake-in)
base_price / qty × scheduledDays  → Trip Services: 실제 일정 일수로 재계산
final_price         → 초과근무(overtime) 아이템
```

`contractedQty = item.quantity`, `scheduledDays` = Trip Services 그룹에서 variant별 실제 스케줄 일수.

### 파트너 결제 Reset 기능 추가
- `deletingPartner` state 추가 — 잘못 입력된 파트너 결제 레코드 삭제 후 재입력 가능
- Reset 버튼 → 파트너 이름 직접 입력 확인 (파괴적 액션 확인 UX 패턴 적용) → `supabase.from('partner_payments').delete().eq('id', existing.id)`

## §4. Partner Payouts 표시 개선

**파일**: `src/app/admin/cases/[id]/page.tsx`

### 금액 원화(KRW) 표시 통일
- "Paid $X of $Y" → "Paid ₩X of ₩Y"
- `fmtUSD(... / exchangeRate)` → `fmtKRW(totalPaid)` / `fmtKRW(totalSuggested)`

### 아이템 단가×수량 분해 표시
qty > 1 이면: `₩단가 × qty단위 = ₩합계` 형식으로 표시 (locked view + unpaid view 양쪽 적용)  
qty = 1 이면: `₩합계`만 표시

예시: `₩1,500,000 × 3d = ₩4,500,000`

## §7. ScheduleEditor — Preview 버튼 + Trip Services 중복 + 정렬 안정화

**파일**: `src/components/admin/ScheduleEditor.tsx`, `src/app/schedule/[slug]/page.tsx`

### 7-1. Preview ↗ 버튼 (Save 없이 바로 미리보기)
- ScheduleEditor 하단에 `Preview ↗` 버튼 추가
- 실제 schedule slug 있으면 `?preview=1&internal=1&draft=1` 붙여 새 탭
- 아직 저장된 schedule 없으면 version=0 sentinel row를 `schedules` 테이블에 upsert → slug 확보 후 열기
- `localPreviewSlug` state로 sentinel slug 세션 내 재사용

**`src/app/schedule/[slug]/page.tsx`**:
- `?draft=1` 쿼리 파라미터 지원 추가
- `cases.schedule_draft_items` 컬럼 읽어서 draft 모드에서 렌더링 (저장된 schedule.items 대신)

### 7-2. 같은 호텔 variant 복수 라인 지원 (itemId)
- `CaseProduct` 타입에 `itemId?: string` 추가
- Trip Services 중복 제거 키를 `variantId` → `itemId ?? variantId` 로 변경
- 같은 SIGNIEL Korean Suite를 남/여 그룹에 각각 예약한 경우 별개 라인으로 표시됨

### 7-3. 편집 중 정렬 튀지 않도록 안정화
- 아이템 정렬 시 `editSnapshots`에 편집 스냅샷이 있으면 원본 기준으로 정렬
- 어드민이 block/time 수정 중에 카드가 다른 위치로 튀는 문제 해결

### 7-4. 호텔 박수 계산 단순화
- 기존: checkout day가 있으면 -1 보정 (복잡한 분기)
- 수정: `max(day) - min(day) + 1` span 기반 단순화
- Day 1 check-in + Day 4 check-out → 4-1+1 = 4박으로 일관 계산
