## 1. `documents.price_rate_snapshot` 마이그레이션 안내

### 증상

```
Could not find the 'price_rate_snapshot' column of 'documents' in the schema cache
```

5/8 작업에서 추가한 `documents.price_rate_snapshot` 컬럼이 Supabase에 반영되지 않은 상태.

### 해결

`sql/2026-05-08_price_rate_snapshot.sql`을 Supabase SQL Editor에서 실행:

```sql
ALTER TABLE documents
  ADD COLUMN IF NOT EXISTS price_rate_snapshot NUMERIC;

INSERT INTO system_settings (key, value)
VALUES ('product_price_rate', '{"usd_krw": 1500}')
ON CONFLICT (key) DO NOTHING;
```

코드에는 `?? 1500` fallback이 있으므로 기존 문서는 자연스럽게 1500원으로 계산됨. 실행 후 PostgREST 스키마 캐시가 자동 reload되며, 안 풀리면 Dashboard → Settings → API → "Reload schema cache" 클릭.

### 부수 사항 — case_contracts realtime

`sql/2026-05-06_realtime_case_tables.sql`(cases / case_contracts / documents / schedules realtime 등록)도 미실행 가능성 있어 함께 점검 권장. 실행되어 있으면 계약서 사인 후 새로고침 없이 반영됨.

## 2. Deposit 플로우 단순화

### 배경

4/30 SOP에서는 deposit이 **두 단계**였음:
1. Agent → Client deposit invoice (client가 agent에게 50% 결제)
2. Admin → Agent deposit settlement (agent가 admin에게 forward)

운영 단순화를 위해 **agent → client 인보이스 발행 단계를 제거**. Agent는 client로부터 오프플랫폼으로 deposit을 받고, admin이 agent에게 deposit settlement만 발행. case status는 settlement leg 하나만으로 advance.

### 변경

**status 전이** ([caseTransitions.ts:61](src/lib/caseTransitions.ts:61))
- `isDepositPaid`는 admin → agent deposit_invoice의 `payment_received_at` 하나만 체크.

**UI** ([CaseDocumentsSection.tsx](src/components/CaseDocumentsSection.tsx))
- Agent의 "+ Deposit" 버튼 제거.
- `IssueIntent`에서 `deposit_to_client` 제거.
- Admin "+ Deposit Settlement" 버튼의 `!has.depositToClient` 선결조건 제거 → 계약 사인 직후 바로 발행 가능.
- 발행/`Mark Paid` 시 agent→client 분기 제거, admin→agent 분기만 유지.
- Agent용 안내 박스 추가(`awaiting_deposit`): quotation × deposit_percentage 추정 금액 + 발행 후엔 실제 settlement total 표시. "이 금액을 client한테 받아서 admin에게 전달하세요" 문구.

**Hero 액션** ([CaseHeroAction.tsx](src/components/CaseHeroAction.tsx))
- AgentCaseHero `awaiting_deposit`: subBits 3개 → 단순 "Forward deposit to admin" 헤드라인.
- AdminCaseHero `awaiting_deposit`: "Agent → Admin" 한 줄.
- Props에서 `depositInvoiceIssued` / `depositPaid` 제거.

**QuoteDocument** ([QuoteDocument.tsx:106](src/components/QuoteDocument.tsx:106))
- final_invoice의 deposit 차감 lookup이 admin→agent settlement 기준으로 변경. Client 뷰에서는 "이미 paid한 deposit = settlement 금액"으로 표시.

**lib helper**
- `issueDepositInvoice` 함수는 backward compat을 위해 남겨두되 호출처 모두 제거. dead code.

## 5. 명칭 정리 (Group vs Shared Activities)

### 배경

기존엔 모든 `document_group`을 UI에서 "Group N"으로 prefix(예: "Group 1: Shared · 5 pax"). Shared는 그룹 개념이 아니어서 어색했음. 또한 케이스의 "인당 그룹"과 헷갈림.

### 결정 (룰)

- **"Group N" 용어 = 인당 묶음(per-person grouping)에만 사용**
- Shared 콘텐츠 = "**Shared Activities**" (그룹 번호 없음)
- Subpackage 묶음 = "**Trip Services**" (그대로)

### 변경

**저장 명칭** — cart에서 'Shared' → 'Shared Activities' 3곳 ([agent/product/page.tsx](src/app/agent/product/page.tsx) lines 155 / 217 / 733). CART_VERSION bump.

**그룹 타입 판별** ([documents.ts:270](src/lib/documents.ts:270))
- `groupTypeFromName`이 legacy `'Shared'`와 신규 `'Shared Activities'` 둘 다 인식 (backward compat). 기존 DB row를 마이그레이트하지 않아도 동작.

**Quotation/Invoice 라벨** ([QuoteDocument.tsx:246](src/components/QuoteDocument.tsx:246))
- Shared Activities → `"Shared Activities · N pax"` (그룹 번호 없음)
- Trip Services → `"Trip Services"`
- 일반 그룹 → `"Group N · X pax"` 또는 `"Group N: customName · X pax"` (유지)

**ScheduleDocument 탭** — `g.name` 그대로 렌더 → 자동으로 "Shared Activities" 표시. 별도 수정 불필요.

**기타** — `admin/cases/[id]` 의 `isSharedGroup: grp.name === 'Shared'` 도 `'Shared Activities'` 추가 인식.

### 4. GuideContent 이미지 업로드 RLS 차단

**증상**: Guide 화면에서 스크린샷/이미지 업로드 시 "업로드 실패: new row violates row-level security policy".

**원인**: `guide` 스토리지 버킷에 INSERT 정책이 없음. `product_images` 테이블 RLS나 `product-images` 버킷 정책은 관계없었음.

**해결** (Supabase SQL Editor):
```sql
CREATE POLICY "Allow authenticated uploads to guide"
ON storage.objects FOR INSERT TO authenticated
WITH CHECK (bucket_id = 'guide');

CREATE POLICY "Allow authenticated updates to guide"
ON storage.objects FOR UPDATE TO authenticated
USING (bucket_id = 'guide');

CREATE POLICY "Allow public read of guide"
ON storage.objects FOR SELECT TO public
USING (bucket_id = 'guide');
```
UPDATE 정책도 필요한 이유: `upsert: true`로 업로드하므로 파일 덮어쓰기 시 UPDATE 경로를 탐.

### 6. Balance Invoice 자동 발행 버그 수정

#### 6-1. 스케줄 재컨펌 시 이전 finalized invoice가 남는 문제

**증상**: 스케줄 revision → 재업로드 → 재컨펌 후에도 이전에 finalize된 `final_invoice.finalized_at`이 남아있어, admin이 Finalize Pricing 없이 balance invoice가 발행된 것처럼 보임.

**수정** ([agent/cases/[id]/page.tsx](src/app/agent/cases/[id]/page.tsx) `confirmSchedule()`):
```ts
// 스케줄 confirm 시 기존 final_invoice finalization 리셋
await supabase.from('documents')
  .update({ finalized_at: null, payment_due_date: null })
  .eq('case_id', caseData.id)
  .eq('type', 'final_invoice')
```
→ 스케줄 재컨펌할 때마다 admin이 항상 가격을 재확인해야 함.

#### 6-2. `createDraftFinalInvoice` 중복 생성 방지 (idempotent)

**증상**: 두 개의 auto-create 트리거(Edit Products + Finalize Pricing 섹션)가 동시에 실행되어 `final_invoice`가 중복 생성.

**수정** ([lib/documents.ts](src/lib/documents.ts)):
- `createDraftFinalInvoice` 호출 전 `final_invoice` 이미 존재하면 기존 것 반환, 신규 생성 안 함.

#### 6-3. `awaiting_pricing` 페이지 로드 시 auto-create 제거

**증상**: admin이 케이스를 열 때마다 `final_invoice`가 없으면 자동 생성 → 삭제해도 계속 재생성.

**수정** ([admin/cases/[id]/page.tsx](src/app/admin/cases/[id]/page.tsx)):
- 자동 생성 스피너 → **"Prepare Balance Invoice"** 명시적 버튼으로 교체
- admin이 직접 클릭해야만 draft final_invoice가 생성됨

#### 6-4. Agent 화면에서 미완성(draft) balance invoice 숨김

**증상**: admin이 Edit Products를 열면 draft final_invoice가 생성되고, agent 화면 INVOICES 섹션에 "Pending" 상태로 노출됨.

**수정** ([CaseDocumentsSection.tsx](src/components/CaseDocumentsSection.tsx)):
```ts
// final_invoice는 finalized_at이 있을 때만 표시
if (d.type === 'final_invoice') return !!d.finalized_at
```
→ Admin이 Finalize 완료 전까지 agent는 balance invoice를 볼 수 없음.
