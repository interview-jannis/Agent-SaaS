### 알림 벨 채널 충돌 디버깅 메모
모바일에서 `<NotificationBell variant="inline" />`, 데스크톱에선 `<NotificationBell variant="floating" />` 둘 다 mount되면 useNotifications가 두 번 호출됨. Supabase realtime은 같은 채널 이름에 두 번 subscribe하면 즉시 reject (`cannot add 'postgres_changes' callbacks for realtime:notifications:UUID after subscribe()`).

해결: useRef로 instance 고유 suffix 생성해서 channel 이름에 포함.

## 2. 데이터 가공 — Selections + Products 이중 모델

### 배경
파트너 이름 노출 정책 검토 중. 사용자: "고객/에이전트에게 partner 노출 X, 어드민만 본다". → **이중 테이블 설계** 필요:
- `selections` = 고객/에이전트 노출용 (anonymized, tier 기반)
- `products` = admin only (실제 파트너 데이터, selection_id FK)

소스 엑셀 `Internal_Price sheet for Proposal_draft_2_v7_26.02.24.xlsx`를 보니 이미 Selection 시트들이 따로 있어서 그 패턴 그대로 가져가면 됨.

| 1. K-Medical | Medical |
| 2. K-Beauty (Medical 라벨) | Beauty |

| 3-5. K-Wellness_Education | K-Education |

### 결정 사항 (오전)
1. **환율 1480**으로 USD→KRW 변환 (필터용만, 저장은 원본 통화 유지)
2. **Medical M/F 분리** (검사 항목이 다르므로 사실상 다른 상품)
3. **VIP 필터 ≥1M KRW** (Hotel은 max 기준)
4. **A/B/C Clinic 익명화 X** — descriptive program name으로 대체 (예: "Premium Stem Cell Anti-Aging · 180m")
5. **매핑 키**: 소스 엑셀의 `Selection Sheet Grade` 컬럼이 이미 매핑 의도로 들어 있음. tier(Silver/Gold/Diamond), 호텔 등급(Deluxe/Suite/Special) 등으로 자동 매칭

### 산출 스크립트
`scripts/build-data-from-master.js` — 단일 스크립트로 두 파일 동시 생성:
- `data/selections_generated.xlsx` — 45개 Selection
- `data/products_generated.xlsx` — 73개 Product (54개 자동 매핑)

**실수**: 처음 빌드할 때 출력 경로를 `products_master.xlsx`로 했다가 사용자가 hand-edit한 파일을 덮어씀. 사용자가 컬럼 삭제/내용 수정한 게 다 사라짐. 사과하고 출력 경로를 `*_generated.xlsx`로 분리. `products_master.xlsx`는 사용자 작업본 영역으로 보존.

### 한국어 → 영어 자동 번역
사용자 요구: "데이터 내 모든 내용은 영어로". 73개 product에 한국어가 산재(특히 호텔 룸 설명, K-Starcation 트레이닝 설명, Vehicle 모델명/조건).

전략:
1. **Phrasebook 기반 치환** — 자주 등장하는 한국어 구문/단어 매핑 (룸구성→Layout, 침실→bedroom, 메리트리무진→Merit Limousine 등)
2. **긴 phrase 우선 정렬** — "동반 시" 같은 multi-word 구문이 "동반" 단어보다 먼저 매칭되도록 길이 내림차순 정렬
3. **잔여 한국어 strip** — phrasebook으로 못 잡은 한국어 hashtag/조사는 정규식으로 제거 (한국어 보존보다 영어-only 일관성 우선)

결과: products / selections 양쪽 모두 한국어 0개.

### 디버깅 — unit 숫자가 가격으로 잡히는 문제
호텔 가격 셀 "~ 13,000,000(세금 별도) / 1박, 2인 기준\n1인당 추가 금액 308,000 / 1박" 같은 경우:
1. "1박", "2인" 같은 unit 숫자가 가격 후보로 들어가서 min이 1로 잡힘 → `stripUnitNumbers` 정규식으로 unit-bound 숫자 제거
2. 그래도 추가 라인의 "1인당 추가 금액 308,000" 때문에 308K가 range로 잡힘 → text 가격은 첫 줄만 사용 (multi-line의 1번째 줄만 본 가격, 나머지 줄은 부가 비용 가정)
3. K-POP Demon Hunters "1Group" 같은 예외 케이스도 추가 → unit 패턴 확장 (group/grp/pax/person/team/팀/그룹 등)

### Hotel 가격 정책
범위가 워낙 넓어서(min~max 2배 이상) "정확한 default" 없음. 결정: `base_price = max` (peak/upper bound). 이유:
- VIP 시장이라 클라가 "할인" 요구 시 admin이 깎는 게 자연스러움 (반대로 min에서 올려야 하면 거부감)
- min/max도 별도 컬럼 보존 → admin이 케이스별 조정

### Vehicle ≥1M 필터
8시간 기준 450K~880K가 대부분이라 1M 필터에 거의 다 걸려 1개만 통과(메리트 리무진 R-VIP 1.32M). 사용자 결정: "그래도 그대로 둬, admin이 비활성화하면 됨".

### 자동 매핑 결과
- 매핑 성공: 54/73 (74%)
- 미매핑 19개: 호텔의 비표준 패키지(Sun Suite Minagi Omakase 등), Beauty의 dermatology 시술(tier가 아닌 시간 기반 메뉴), K-Education 일부
- 미매핑 품목은 admin이 UI에서 수동으로 selection 매칭

## 3. 미팅 후 — 전제 변경 (오후)

→ **파트너 이름 공개 OK**. 위에서 한 이중 모델 작업의 핵심 동기(익명화)가 사라짐.

### 영향
1. **Selections 모델 폐기 확정** (사용자 결정)
2. **단일 products 테이블**로 통합. 각 product에 WHY/WHAT/WHO 컨텍스트 추가하는 방향
3. partner_name 컬럼은 그대로 노출 (admin/agent/customer 모두에게)
4. 이미 만든 `data/selections_generated.xlsx`는 reference 보관, DB import 안 함 — 추후 삭제 예정
5. `scripts/build-data-from-master.js`도 향후 products 부분만 남기고 selections 추출 로직 제거 예정

### 데이터 가공 신규 요구사항
각 상품마다 다음 컨텍스트가 들어가야 함:
- **Why**: 왜 이 상품/기관인지 (예: "한국 최고 프리미엄 대학병원", "가성비 좋음", "최첨단 시술 테크놀로지")
- **What**: 상품 자체 설명 (현재 description 칸으로 충분)
- **Who**: 대표 원장 프로파일 (영문 오픈 정보 사용, 직접 시술자 X)

→ 스키마 / 엑셀 포맷에 컬럼 추가 필요: `why_recommendation` (TEXT), `head_doctor_profile` (TEXT or JSONB)

### 일괄 업로드 기능
사이트에서 1건씩 등록 외에 **엑셀 일괄 업로드** 기능 신규 요구. 정해진 포맷으로 엑셀 던지면 자동으로 카테고리 분류 + admin 등록 + agent쪽 자동 매칭. 우리 build-data 스크립트와 사실상 같은 일을 사이트 내에서 admin이 직접 수행.

→ products import API 신규 + Admin Products 페이지에 import 버튼 추가 작업.

- 오전에 6시간 들여 Selection/Products 이중 모델 만들었는데 오후 미팅에서 "익명화 안 해도 된다"는 결정. 이 작업 자체가 무용은 아니지만(Selection 패턴은 grouping 용도로 살릴 수 있음), 가장 핵심 동기가 사라짐.
- → 큰 아키텍처 결정은 stakeholder 확인을 한 박자 먼저 받는 게 맞다. 상위 결정 안 잡혔는데 구현부터 들어가면 시간 손실.

**"엑셀 자료 업로드 → 자동 분류"의 가치**:
- 상품 등록을 사이트 내에서 1건씩이 아니라 일괄 처리하는 방향 → MVP 단계에서 데이터 입력 효율 압도적. 우리 build-data 스크립트와 같은 일을 admin이 사이트에서 직접 하는 형태. invoice 자동 생성 패턴이랑 일관.

**스케줄 PDF → 링크 트렌드**:
- "PDF 메일로 주고받는 시대 지났다, 모든 데이터 공유는 링크 컨셉으로". 이미 quotation/invoice는 링크 기반 — 스케줄도 같은 패턴으로 통일.

**파트너 익명화 정책 변경의 교훈**:
- 외부 노출 vs 내부 정보 구분이 처음에 필요하다고 가정했는데, 실제로는 "VIP는 어차피 한국 와서 다 알게 됨". 비즈니스 모델의 실체를 충분히 파악 못한 채 데이터 모델 설계로 들어간 셈. 다음부터는 "고객이 결제 끝나고 나면 어떻게 알게 되나?" 시나리오를 먼저 그려야 함.
