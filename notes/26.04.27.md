## 2. 흐름 B 적용 — 결제↔스케줄 순서 뒤집기

### DB 마이그레이션
```sql
-- ENUM → TEXT + CHECK (CLAUDE.md 권장 방식)
ALTER TABLE cases ALTER COLUMN status DROP DEFAULT;
ALTER TABLE cases ALTER COLUMN status TYPE TEXT;
DROP TYPE case_status;
UPDATE cases SET status = 'quote_sent' WHERE status = 'payment_pending';
ALTER TABLE cases ALTER COLUMN status SET DEFAULT 'quote_sent';
ALTER TABLE cases ADD CONSTRAINT cases_status_check
  CHECK (status IN ('quote_sent', 'schedule_reviewed', 'schedule_confirmed', 'payment_completed', 'travel_completed'));
```

### 새 흐름
| 상태 | 라벨 | 색 | 의미 |
|---|---|---|---|
| `quote_sent` | Awaiting Schedule | amber | 견적 발송, admin 스케줄 작업 대기 |
| `schedule_reviewed` | Schedule Reviewed | violet | admin 업로드, agent 검토 대기 |
| `schedule_confirmed` | Awaiting Payment | blue | agent 컨펌, 가격 finalize → 결제 요청 |
| `payment_completed` | Payment Confirmed | emerald | 결제 확정, 여행 대기 |
| `travel_completed` | Travel Completed | gray | 종결 |

### 코드 변경 (~14파일)
- 일괄 sed: `payment_pending` → `quote_sent`, 라벨/색 매핑
- 의미 변경: schedule_confirmed = "Awaiting Payment", payment_completed = emerald
- transition 로직 재배열:
  - 케이스 생성: quote_sent
  - 스케줄 업로드: quote_sent || schedule_reviewed(revision) → schedule_reviewed
  - 스케줄 컨펌(agent): schedule_reviewed → schedule_confirmed
  - 결제 확정(admin): schedule_confirmed → payment_completed
  - 여행 완료(agent): payment_completed → travel_completed
- Cancel Case: 결제 전(quote_sent / schedule_reviewed / schedule_confirmed)만 가능
- Mark Travel Complete: payment_completed에서만 (이전엔 schedule_confirmed)
- Pipeline 컬럼 순서 / Action Required 큐 매핑 / 알림 문구 모두 재배열

### 흐름 B의 새 단계 — Finalize Pricing
미팅에서 짚은 "스케줄 컨펌 → 가격 수정 → 결제 요청"을 코드로:
- DB: `quotes.finalized_at`, `quotes.invoice_number`
- Admin Case 상세: schedule_confirmed에서 quote_items.final_price 인라인 편집 + Save
- Save 시: total_price 갱신, finalized_at 셋, invoice_number 첫 발번 (`#INV-001`)
- Save 후 Confirm Payment 블록 활성화
- Edit pricing 가능 (재수정 시 알림 문구 다름)

## 3. Quotation vs Invoice 분리

### 명칭/번호
- `#Q-001` (quote_number, 항상 유지)
- `#INV-001` (invoice_number, finalize 시 자동 발번 — 기존 finalized 카운트 +1)
- 듀얼 넘버링: 한 record + 두 번호 (Option B)
- 향후 회계감사 필요해지면 별도 invoices 테이블로 마이그레이션 가능 (스냅샷 컬럼 박아두면)

### `/quote/[slug]` 양식 분기
| | Quotation | Invoice |
|---|---|---|
| 헤더 | "Quotation" (그린) + "Estimated · Subject to confirmation" | "Commercial Invoice" (그레이) |
| 디스클레이머 | amber 배너 (스케줄 컨펌 후 final invoice 발급 안내) | 없음 |
| Ref No | `#Q-001` | `#INV-001` (+ "Quote Ref: #Q-001" 별도 줄) |
| 마감일 | 표시 안 함 (스케줄 조율 기간 가변) | "Due Date" 표시 |
| 총액 라벨 | "Estimated Total" | "Total Amount" |
| 본문 | "share this quotation... finalize once schedule is confirmed" | "submit an invoice..." |
| **은행 계좌** | **숨김** | 표시 |

### 강제 모드
- `?as=quotation` / `?as=invoice` URL 파라미터로 강제 렌더
- Admin Case 상세 finalize 후엔 Quotation/Invoice 두 링크 (둘 다 미리보기 가능)
- Agent는 현재 상태만 (혼동 방지) — 클라이언트 보낼 때 명확

### 알림 + 추적
- `invoice_first_opened_at` 컬럼 — invoice 단계 첫 열람 추적
- 가격 수정(reprice) 시 invoice_first_opened_at 리셋 → 다음 클라이언트 열람 시 다시 알림
- 알림 메시지에 `#INV-001` 사용 (이전엔 `#Q-001`로만 떴음 — 버그)
- 분기:
  - 견적 단계 첫 열람 → "Quotation opened by client"
  - Invoice 단계 첫 열람 → "Invoice opened by client"
  - Finalize 첫 시 → "Pricing finalized — invoice ready to send"
  - Reprice 시 → "Invoice pricing updated — please review before resending"

### Agent 버튼 동적 라벨
- finalized 전: **Send Quotation**
- finalized 후: **Send Invoice**
- 기준이 case status가 아니라 `quote.finalized_at` (가격 수정 후 자동 전환)

## 4. 정산 흐름 정밀화

### Partner Payouts (신규)
DB:
```sql
CREATE TABLE partner_payments (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  case_id UUID NOT NULL REFERENCES cases(id) ON DELETE CASCADE,
  partner_name TEXT NOT NULL,
  amount NUMERIC NOT NULL,
  paid_at DATE NOT NULL,
  paid_by UUID REFERENCES admins(id) ON DELETE SET NULL,
  note TEXT,
  created_at TIMESTAMPTZ DEFAULT now(),
  UNIQUE (case_id, partner_name)
);
```

UI (Admin Case 상세 Partner Payouts 섹션):
- 견적 상품의 `partner_name` 자동 그룹핑
- 각 파트너별: 추천 금액(`Σ base_price`) + Paid On + Note 입력
- 상품별 breakdown 표시 ("Smith Family · 2 pax")
- 결제 전 잠김 (`payment_completed` 이전엔 disabled)
- Edit/Delete 가능, audit log 기록

### Agent Settlement (Case 상세에 추가)
- 기존 Settlement 페이지에만 있던 settle 버튼 → Case 상세에도
- travel_completed 시점에만 Mark Paid 가능
- 자동 commission 계산 (수정 불가, audit 정합성)

### Admin Settlement 페이지 좌우 분할
이전: 통합 Hero + sparkline + Pending + History (정신없음)
새 구조:
```
┌─ 🏥 Partner Payouts (slate)  ─┬─ 👥 Agent Settlements (amber) ─┐
│  Mini Hero (3 KPIs)           │  Mini Hero (3 KPIs)            │
│  Pending 테이블                │  Pending (agent별 그룹)        │
│  History 테이블                │  History 테이블                │
└───────────────────────────────┴─────────────────────────────────┘
```
- 좌측 컬러 띠 + 아이콘만 색 강조, 안쪽은 중립 gray
- xl 브레이크포인트(1280px+)에서 좌우, 그 이하는 스택

### Cash basis 통일
- Hero "Agent Margin" → **Agent Payouts** (settlements.paid_at 합산, cash basis)
- Partner Cost: 이미 `partner_payments.paid_at` 합산 (cash basis)
- Earnings는 발생주의(고객 결제 시점), Revenue는 cash IN
- 4지표 의미 명확: Revenue/Earnings는 매출 인식, Partner/Agent는 cash out

### Cases 리스트 정산 상태 뱃지
- Admin Cases: `[P●][A●]` 미니 도트 — emerald(done)/amber(pending)/gray(N/A)
- Agent Cases: travel_completed 케이스만 — Received(emerald)/Pending(amber) 뱃지

## 느낀 점 / 메모

- **흐름 B는 작업량보다 영향 범위가 핵심**. sed로 라벨 일괄 변환 후 transition 로직 손보는 게 무거움. 14파일이라 했지만 실제로 영향 받는 곳은 더 많음. 다행히 한 호흡에 끝남.
- **Quotation vs Invoice 분리는 4/24 미팅의 진짜 가치**. 단순 라벨이 아니라 "어떤 단계의 문서냐"가 양식·번호·은행 표시·문구 다 결정. 코드로 옮기면서 비즈니스 명료성도 같이 올라감.
- **Partner cash basis 추적은 Hero "거짓말" 해소**: 이전엔 Partner Cost가 견적상 base 합 (실제 송금 X). 이제는 진짜 통장 나간 돈만 잡힘. Admin이 "병원에 얼마 보냈나" 정확히 답할 수 있음.
- **Sparkline은 데이터 없으면 의미 없다**: 오늘 데이터 와이프 후 모두 0으로 보임. 시뮬레이션 5/11~15에서 데이터 쌓이고 나야 진가 발휘.
- **다크모드 가드**: 사용자 PC 다크모드 켜져있어 폼 input들이 회색으로 보였음. 단순한 CSS 한 줄(`color-scheme: light`)이 모든 input 다크 변환 차단. 작은 디테일이지만 시뮬레이션 환경에서 큰 차이.
- **사용자가 "왜 이 가격이지?" 의문을 갖는 순간 신뢰 깨짐**: Partner Payouts에 상품/그룹/인원 breakdown 추가 — admin이 송금할 때 "어떤 명목" 즉시 알 수 있음. 이런 디테일이 시뮬레이션에서 차이 만듦.
