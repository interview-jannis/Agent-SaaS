## 1. Dashboard 티어 지표 — "당월 환자 수"

### 문제
- 기존: `cases.filter(travel_completed).length` — 누적 케이스 수
- 실제 마진율 스펙(CLAUDE.md): 당월 여행완료 "**고객 수**" 기준 (0-10: 15%, 11-30: 20%, 30+: 25%)
- 브랜드 자료: `10 / 30 Patients/Month` (케이스 아님, 환자임)

### 수정
- 단위: cases → **case_members.length 합산** (그룹 4명 케이스 = 4명)
- 기간: 누적 → **당월** (`travel_completed_at.startsWith(monthKey)`)
- 임계값: 11/31 → **10/30** (브랜드 자료 기준)
- 라벨: "N / 11 cases" → **"N / 10 patients this month"**

### DB 추가
```sql
ALTER TABLE cases ADD COLUMN travel_completed_at TIMESTAMPTZ;
UPDATE cases SET travel_completed_at = COALESCE(travel_end_date::timestamptz, now())
WHERE status = 'travel_completed' AND travel_completed_at IS NULL;
```
Mark Travel Complete 핸들러에서 자동으로 `new Date().toISOString()` 기록.

### 교훈
select에 없는 컬럼 참조하면 PostgREST가 400 반환 → cases 빈 배열 fallback → 화면 다 사라짐. 컬럼 추가 시 select문과 type도 같이 반드시 업데이트.

## 2. DateTime24Picker 공용 컴포넌트

Chrome on Windows가 `lang="en-GB"` 무시하고 OS 로케일 따라가서 datetime-local이 12h로 나오는 문제. 커스텀 컴포넌트로 해결:

- `<input type="date">` + 시(00-23 select) + 분(00-59 select) 3칸
- 내부 state로 partial 선택 유지 (DOBPicker에서 겪은 "3개 안 고르면 보임 안 됨" 버그 회피)
- `minDate` prop — case.created_at 이전 날짜 선택 불가

Agent Cases 상세의 Outbound/Inbound 4개 datetime 모두 교체.

## 5. 계약서 시스템

### 법적 용어 재정리
사용자 지적: Tiktak은 **제품 브랜드**, 법적 계약 당사자는 **Interview Co., Ltd.**
- 기존 초안: `Interview Co., Ltd. ... ("Tiktak")` → 이후 본문에서 "Tiktak shall..."
- 수정: `Interview Co., Ltd. (the "Company"), which operates the Tiktak platform` → 본문은 "the Company shall..."
- Tiktak은 단순 제품명으로만 사용 (정의어 아님)

### 토큰 치환
- 템플릿 body에 `{{AGENT_NAME}}`, `{{AGENT_COUNTRY}}` 토큰
- 서명 시 ContractStep이 실제 값으로 치환 후 body_snapshot 저장 (audit 보존)
- `**{{AGENT_NAME}}**` (볼드 감싸도 됨) — 더블스타 중복 방지 regex 포함

### Markdown 렌더
- `## heading` → h2/h3
- `- item` 연속 → ul/li
- 빈 줄 → 문단 구분
- `**bold**` → `<strong>` (`***`, `****` 과다 스타 → `**` 정규화)
- 문단 내 하드랩된 줄은 space-join (source 줄바꿈 무시)
- 간격: 제목↔본문 tight, 문단↔다음제목 wide (섹션 시각 분리)

### 계약 중 사용자 동선
1. 계약서 **먼저 읽음**
2. 하단에 "Your Information" 인풋 (이름/국적) — NDA에서만
3. 체크박스 + 서명 캔버스
4. 제출 → body_snapshot에 치환된 본문 저장 + agents 테이블에 name/country 업데이트

Partnership 페이지는 "Signing as {name} · {country}" 표시만 (NDA에서 저장된 값 사용).

### Admin 계약 뷰어 (`/admin/agents/[id]/contract/[contractId]`)
- 공식 문서 스타일 레이아웃 (제목 bar, 당사자 메타, 본문, 서명 이미지, audit footer)
- `Print / Save PDF` 버튼 + `print:hidden`으로 네비 숨김 → 브라우저 인쇄로 PDF 저장
- 이메일(temp) 같은 계약 무관 필드는 뷰어에서 제외

### Agent 본인 계약 뷰어 (`/onboarding/contract/[contractId]`)
- 본인 소유 계약만 열람 (다른 agent contractId 요청 시 "Contract not found" — 보안 가드)
- Waiting 페이지에 "Your Signed Agreements" 리스트 + 클릭 열람

## 6. 온보딩 플로우 완성

### 흐름
```
[Admin] +Create Temp Agent
  ↓ (auth_users 중복 회피 — tempNN 자동 증가)
Temp login (temp01/temp01)
  ↓ /onboarding
Welcome → Orientation(PDF, 1/3) → NDA(identity+sign, 2/3) → Partnership(sign, 3/3)
  ↓ 서명 완료 → agents.onboarding_status = 'awaiting_approval'
  ↓ notifyAllAdmins("#AG-005 signed contracts — review needed")
Waiting 화면 (15s 폴링 — "Check status" 버튼 제거)
  ↓ Admin 승인 시 자동 리다이렉트
Setup Wizard (실제 이메일/비번 변경 + Bank 필수 + Phone 선택)
  ↓ setup_completed_at 기록
Agent Home 정상 진입
```

### Setup Wizard API (`/api/agent/setup`)
- Service role로 `auth.admin.updateUserById` 호출 → email 변경 시 confirmation flow 우회
- agents 테이블 email/phone/bank_info 업데이트 + setup_completed_at 기록
- 검증: 비번 8자 이상, email !endsWith('@tiktak.temp'), bank 4필드(Bank/Account/Beneficiary/Swift) 필수

### 라우팅 가드
- `AgentOnboardingGuard` (agent layout): approved + setup_completed 확인, 아니면 `/onboarding` 또는 `/onboarding/setup` 리다이렉트
- `Onboarding Layout`: approved+setup 완료되면 `/agent/home`으로 리다이렉트 (onboarding 재진입 차단)
- `Login` 페이지: 로그인 후 onboarding_status + setup_completed_at에 따라 3-way 분기

### Admin 승인 UI
- `/admin/agents` Onboarding 섹션: `awaiting_approval`만 클릭 가능(상세 진입)
- `/admin/agents/[id]`: **Temp 계정엔 Metrics/Profile/Bank/Cases/Settlement 숨김** (채울 수 없는 필드)
- Signed Contracts 섹션 + Approve & Activate 버튼 (contracts.length < 2면 비활성화)
- Approve 시 `onboarding_status=approved`, `is_active=true`, `agent_contracts.approved_at/approved_by` 기록

### Temp 생성 API 개선
- 기존: `agents.count + 1` → agents 지웠는데 auth.users에 남아있으면 충돌
- 수정: `auth.admin.listUsers`로 스캔해서 점유된 번호 skip

### 에러 학습
- `agent_contracts.approved_by` → `admins.id` FK가 DELETE NO ACTION이라 admin 삭제 불가
- 해결: `ON DELETE SET NULL` — audit은 유지, 승인자만 NULL

```sql
ALTER TABLE agent_contracts DROP CONSTRAINT agent_contracts_approved_by_fkey;
ALTER TABLE agent_contracts ADD CONSTRAINT agent_contracts_approved_by_fkey
  FOREIGN KEY (approved_by) REFERENCES admins(id) ON DELETE SET NULL;
```

## 7. Realtime 알림 시스템

### DB
```sql
ALTER TABLE notifications ADD COLUMN IF NOT EXISTS created_at TIMESTAMPTZ DEFAULT now();
ALTER TABLE notifications ADD COLUMN IF NOT EXISTS link_url TEXT;
ALTER TABLE notifications DISABLE ROW LEVEL SECURITY;
GRANT ALL ON notifications TO anon, authenticated;
ALTER PUBLICATION supabase_realtime ADD TABLE notifications;
```

### 구조
- `src/lib/notifications.ts`: `createNotification`, `notifyAgent(agentId, msg, link)`, `notifyAllAdmins(msg, link)` (auth_user_id 기준 dedup)
- `src/hooks/useNotifications.ts`: INSERT 이벤트 subscribe, unread 카운트, markAllRead
- `src/components/NotificationBell.tsx`: 우측 하단 fixed, 빨간 점 unread 표시, 클릭 열기, 각 행 click → `link_url`로 이동

### 알림 UX
- 기본: **안 읽은 것만 표시**
- 열면 즉시 markAllRead, "방금 읽음" 아이템은 세션 내 유지 (열자마자 빈 화면 방지)
- 하단 `Show previously read (N)` 토글로 히스토리 접근

### 트리거 9개 wiring
**Agent 받음 (4):**
1. Schedule uploaded — Admin case 상세 업로드 시
2. Payment confirmed — Admin이 Confirm Payment 시
3. Settlement paid — Admin Settlement 모달 Confirm 시
4. Invoice opened by client — (서버 컴포넌트 미구현, 남은 작업)

**Admin 받음 (5):**
1. New case created — Agent review에서 quote insert 완료 시
2. Schedule v{N} revision requested — Agent가 revision 요청 시
3. Travel completed — Agent가 Mark Travel Complete 시
4. Schedule v{N} confirmed — Agent가 Confirm Schedule 시
5. `#AG-00N` signed contracts — 마지막 계약(Partnership) 서명 시 (이름 대신 agent_number — temp 계정 찾기 쉬움)

### 이슈 해결
- 중복 알림: admins 테이블에 동일 auth_user_id 2row 있으면 2개 insert → `notifyAllAdmins`에 Set 기반 dedup
- Schedule upload 알림 누락: Case select에 `agent_id` FK 컬럼 직접 포함 안 해서 `caseData.agent_id` undefined → select에 명시적 추가

## 9. Agent Home Cart 복원

### 문제
`Create Quote` → Review → `Edit` 눌러 Home 돌아오면 선택한 상품이 초기화됨.

### 고민
- Review에서 Edit 복귀 시엔 복원해야 함
- 그런데 그냥 localStorage 복원하면 다른 탭 이동했다 와도 복원돼서, 실수로 담고 버린 cart가 귀신처럼 살아남음

### 해결: sessionStorage 플래그 방식
- Review의 Edit 버튼: `sessionStorage.setItem('agent-cart-restore', '1')` + router.back()
- Home mount: 플래그 있으면 복원 + 플래그 제거, 없으면 localStorage.removeItem

시나리오별:
- Home → Create Quote → Review → Edit → Home ✅ 복원
- Home → Create Quote → Review → Cases → Home ✅ 클리어
- Home → Cases → Home ✅ 클리어
- 새 로그인/새로고침 ✅ 빈 상태

## 느낀 점 / 메모

- **Token 치환은 audit과 양립 어려움**: body_snapshot에 치환된 값 저장 → 에이전트 이름이 나중에 바뀌어도 계약서는 원본 유지. 맞는 방향이지만, 치환 실수(e.g. `**{{TOKEN}}**` vs `{{TOKEN}}` double-bold) 추후 발견돼도 재렌더할 수 없음. 템플릿 쓰기 시점에 검증 로직 들어가면 좋겠음.
- **Realtime은 생각보다 쉬웠음**: Supabase Realtime publication 한 줄 + client subscribe 5줄. 30초 폴링 대비 설정 부담 거의 없음. 앞으로 상태 변화 UX는 기본으로 Realtime 가정.
- **플래그 기반 상태 복원 (sessionStorage)**: cart 복원 문제 해결 방식이 깔끔했음. 단순 "이전 상태 저장"보다 "의도적 이벤트에만 복원" 패턴이 UX 기대에 더 부합.
- **Temp agent onboarding_status FSM 정교화 필요성**: pending → awaiting_approval → approved(setup pending) → approved(setup done) 4단계 정확히 라우팅 가드해야 새로고침/깊은 링크 다 막음. 한 군데(Login/Guard/Layout) 동기화 실수 만나면 리다이렉트 루프나 엉뚱한 화면 보임. 3곳 전부 동일 로직 유지하는 게 핵심.
- **계약서는 법무 검토 필수 메모**: 초안에 Tiktak을 "Company" 자리에 넣었던 실수처럼, 저 같은 non-legal이 쓰는 계약서는 대체로 틀리다고 봐야 함. MVP 전에 법무팀/변호사 검토 꼭.
