### 1. 케이스 페이지 상태별 그린 강조 정리 (admin + agent)

상태 전이 12단계 전부 훑으면서 각 섹션의 "내 차례인가" 강조 룰을 단일 진실 소스로 정렬.

- **Admin Schedule History**: `awaiting_schedule / reviewing_schedule`에 더해 `awaiting_travel`도 active로 인식 (스케줄 컨펌 후에도 여행 전까지는 admin이 "확정 운영 필드" 편집하는 단계니까).
- **Admin Reviews 섹션**:
  - render 조건 확장: `caseSurvey || awaiting_review / awaiting_settlement / completed`. 이전엔 survey 없으면 안 보였는데 이제 `awaiting_review`에서도 "Awaiting client review submission." placeholder 노출.
  - `awaiting_review / awaiting_settlement`일 때 헤더 그린 강조.
- **Admin no-contract 상태 색상**: `AdminCaseContractSection`의 계약 없음 분기가 awaiting_contract여도 회색이던 것을 그린(white bg + `#0f4c35` border)으로 일관화.
- **Admin financialStages**: 그린 보더 유지 단계에서 `awaiting_pricing` 제거 + `awaiting_review` 추가 (pricing 단계는 별도 Pricing 섹션이 강조, Financials는 결제·여행·정산까지 캐리).
- **Agent financialStages**: `awaiting_travel`, `awaiting_review` 추가.
- **Agent Trip Setup / Trip Info / Members / member checklist**: 강조 조건을 `infoCollectionActive` (= 디파짓·계약 외 거의 모든 단계)에서 `caseData.status === 'awaiting_info'`로 좁혔다. 이전엔 awaiting_deposit 이후에도 trip setup이 빨갛게 강조돼서 시각적으로 시끄러웠음.
- **Admin Schedule placeholder** (`sortedSchedules.length === 0`): `awaiting_schedule`일 때 그린 보더 + 그린 헤더 텍스트로 강조 (이전엔 회색만).

### 2. Hero 글로벌 sticky 리프트

문제: `awaiting_schedule` 단계에서 Hero 박스가 스케줄 영역 진입하면 sticky가 풀리고 ScheduleEditor 내부 하단 toolbar(+ Add Item / + Free time / + Prayer / + Fill Transfers)에 가려졌음.

원인:
- Hero가 `max-w-3xl pre-schedule` 래퍼 안에 있었고 그 래퍼가 스케줄 섹션 시작 전에 닫혀 sticky 부모가 사라짐.
- Hero `z-20` + ScheduleEditor bottom toolbar `z-20` 동등 → DOM 순서상 toolbar가 위.

해결:
- Hero를 outer flex 컨테이너(`<div className="w-full px-4 md:px-6 py-4 md:py-6 flex flex-col gap-5">`) 직속 child로 이동. sticky 부모가 전체 스크롤 컨테이너 → 페이지 끝까지 Hero 유지.
- 내부에 `max-w-3xl mx-auto w-full` 래퍼만 다시 두고 가로폭 그대로.
- `z-10` → `z-30` 상향. ScheduleEditor 하단 toolbar `z-20`보다 위.
- `order-first` 제거 (DOM 순서상 이미 최상단).

### 3. Add Item 그룹 강제 선택

문제: 새 item 추가 시 group이 자동으로 `Shared (all)` 선택돼서 admin이 그룹 의식 없이 저장하는 사례 발생.

해결 (`ScheduleEditor.tsx`):
- pending 버튼 라벨 변경: `isGroupUnset` 시 "Shared (all)" → "— Choose group —" (amber 톤).
- `canSave` 가드 강화: `unsetGroupItemIds.size === 0` 추가.
- 미선택 시 amber 경고 박스 + Save 버튼 disabled + tooltip "Choose a group for every item first".

### 5. Hotel sub-dropdown 3-enum 축약 (`check-in / stay / check-out`)

기존: `checkin | checkout | stay | depart | return` 5-enum (`depart/return`은 레거시 hotel bookend 기능 잔재).

변경 (`src/types/schedule.ts`):
```ts
hotelCheckType?: 'checkin' | 'checkout' | 'stay' | null
```

- 드롭다운 UI는 이미 3개로 좁혀져 있었음 → type만 정리.
- 타입 분기 로직(`onChange`에서 자동 title 채움)의 4-enum 시제 제거 → checkin/checkout/stay 3분기.
- 사용자 합의 룰: 도착일 evening = check-in, 중간일 morning = stay (Fill Transfers의 anchor), 마지막날 morning = check-out, evening 호텔 = stay. customer view에 morning stay 노출.

### 6. Hotel 박수 카운트 — 첫 등장 ~ 마지막 등장 (day span)

문제: Trip Services 카운트는 "unique day 수"인데 hotel은 의미가 달랐음. Day 1(check-in)과 Day 3(check-out)만 잡으면 2일로 카운트 → 실제론 3일 staying.

해결 (`ScheduleEditor.tsx`, coverage gate):
- `cp.isHotel` 분기에서 `max(day) - min(day) + 1` 로 span 계산.
- 그 외 trip services는 unique day 수 유지.

### 7. Subpackage selector에서 호텔 제외

`uniqueTripServices = caseProducts.filter(cp => cp.isTripService && !cp.isVehicle && !cp.isHotel)`로 좁힘. hotel은 자체 appointment(`hotelCheckType` 포함)이고 vehicle은 transfer에 붙는 별도 picker가 있으니 concierge subpackage selector에서 둘 다 빠짐.

### 8. Day-level concierge subpackage 신규 도입 (가장 큰 변경)

**배경:** Trip Services 카운트가 item별 `tripServiceVariantIds`에 의존 → admin이 매 item마다 컨시어지 칩 토글하는 작업이 부담. 같은 컨시어지가 하루 종일 따라다니는 게 일반적이라 "Day 단위 default" 모델로 전환.

**저장 모델:**
- `schedules.day_subpackages JSONB DEFAULT '{}'::jsonb` (신규 컬럼, 마이그레이션: `sql/2026-05-15_schedules_day_subpackages.sql`)
- 구조: `{ "1": ["variantId-A", "variantId-B"], "2": ["variantId-A"] }`

**ScheduleEditor 변경:**
- 신규 prop `initialDaySubpackages?: Record<number, string[]>`
- state + `setDayVariants` / `toggleDayVariant` 헬퍼
- Day 카드 콘텐츠 상단 박스에 "Day concierge" 라벨 + 토글 chip 행 (hotel/vehicle 제외한 trip services).
- 카운트 로직 재설계:
  - **호텔**: 기존 day span 유지(첫~마지막 등장).
  - **호텔 외 trip services**: 카운트의 1차 소스는 **day-level `daySubpackages`**. item의 `variantId` 직접 매칭(예: vehicle linked transfer)도 합산.
  - item-level `tripServiceVariantIds`는 informational override로 남기되 카운트엔 반영 X.
- 저장 흐름: `schedules.insert(...)`에 `day_subpackages` 추가.

**Admin 페이지 (`admin/cases/[id]/page.tsx`):**
- `schedules(... day_subpackages)` select에 추가
- `<ScheduleEditor initialDaySubpackages={...}/>` 주입

**Q5 합의된 디테일:**
- (Q1) JSONB 새 컬럼 / (Q2) Day 헤더 인라인 / (Q3) Replace 의미 / (Q4) day-level만 카운트 / (Q5) hotel·vehicle 둘 다 제외.
- 기존 케이스는 전부 시뮬 데이터라 마이그레이션 영향 없음.
