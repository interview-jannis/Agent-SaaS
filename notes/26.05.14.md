### 2. ScheduleEditor 그룹 컬럼 레이아웃

**기존**: 한 day 카드 내 모든 항목이 세로로 나열. 그룹별 구분은 좌측 색띠 + chip만.

**변경**: caseGroups가 있으면 항목을 segment로 분할 후 column 기반 렌더:
- `Seg = { type: 'shared'; item } | { type: 'group'; items[] }` 빌드
- **shared segment (groupIds=null)**: 항상 full-width row, `bg-green-50/30` + `border-l-2 border-[#0f4c35]/20`
- **group segment (groupIds 있음)**: 행 기반 렌더, group 컬럼별 cell. 컬럼 헤더 행 (`Mom`, `Dad`, `Kids` 등) 최상단에 sticky 헤더.

#### 2-1. 그룹 컬럼 셀 스팬 (인접 multi-group)

같은 `item.id`가 인접한 컬럼들에 할당돼 있으면 → cell을 합쳐서 (`flex: span 1 0`) 한 번만 렌더.

- 예: `groupIds=[Mom, Dad]` (인접) → Mom, Dad 컬럼 너비를 합쳐 1개 cell로 렌더
- 동일 row에서 Kids만 다른 item이면 Kids는 자기 컬럼 너비 그대로
- 한쪽 그룹에 item이 모자라면 그 자리에 `—` placeholder

```tsx
// 행 i의 cell 빌드 (consecutive same-id 그룹 머지)
while (i < colRow.length) {
  const item = colRow[i]
  if (item === null) { cells.push({ item: null, span: 1, ... }); i++ }
  else {
    let span = 1
    while (i + span < colRow.length && colRow[i + span]?.id === item.id) span++
    cells.push({ item, span, ... })
    i += span
  }
}
```

#### 2-2. 비인접 그룹 처리 (subset shared banner)

그룹 컬럼 순서가 `[Mom, Dad, Kids]`일 때 `groupIds=[Mom, Kids]`는 비인접 — 셀스팬으로 합칠 수 없음. 그대로 두면 같은 item이 분리된 2개 cell로 렌더되어 Edit/Remove 버튼이 두 군데 → admin UX 사고.

**해결**: 같은 row 안에서 unique item의 column 위치를 수집해 contiguous 여부 판정. 비인접이면 그 item을 row에서 빼서 **full-width banner row**로 위에 띄워 표시:

- banner row: `bg-emerald-50/40` + `border-l-2 border-emerald-400/40` — shared row와 다른 톤
- ItemRow 내부의 기존 `Covers Mom, Kids` chip이 어떤 그룹을 묶었는지 자연 표시
- column row에선 비인접 item의 자리가 placeholder로 비고, 동일 row의 다른 인접/단일 그룹 item들은 컬럼에 그대로 남음
- column row가 전부 비면 통째로 skip (banner만 노출)

```tsx
const positionsById = new Map<string, { item; positions[] }>()
row.forEach((it, idx) => { if (it) positionsById.get(it.id)?.positions.push(idx) ?? ... })
const bannerItemIds = new Set<string>()
positionsById.forEach(({ positions }, id) => {
  if (positions.length <= 1) return
  const sorted = [...positions].sort((a, b) => a - b)
  if (sorted[sorted.length - 1] - sorted[0] !== sorted.length - 1) bannerItemIds.add(id)
})
```

대안 검토했지만 기각:
- **컬럼 동적 재정렬** → Mom 컬럼이 row마다 위치 바뀜, 매우 혼란
- **비인접 multi-group 차단** → admin에게 그룹 순서를 강제하게 됨, 과도한 제약
- **현재 그대로 두기 (2개 cell 분리 렌더)** → 같은 item 편집 버튼 두 개, admin이 어떤 게 본체인지 혼란

### 3. Hotel → Appointment 흡수

**기존**: `itemType='hotel'`이 별도 선택지. 호텔 작업은 (1) Hotel 타입 고르기 → (2) Check-in / Check-out 종류 정하기 → (3) Stay row는 별도로 추가, 단계 많고 분기된 흐름.

**변경 (`itemType` 선택지 단순화)**:
- 선택지에서 hotel/free 제외 (`SCHEDULE_ITEM_TYPES = ['appointment', 'transfer', 'meal']`)
- 타입 union에는 `'hotel' | 'free'` 남아 있어서 **legacy row 읽기는 가능** (backward compat)
- `addItem` 기본값은 'appointment'

**Hotel은 appointment 안에서 처리**:
- appointment dropdown product picker에 호텔 product 노출 (`pickerProducts`에서 `cp.isVehicle`만 제외, hotel은 통과)
- 호텔 product 선택 시 `selectedHotelProduct`로 감지 → **Check-in / Check-out / Stay** 서브 드롭다운 자동 노출
- 서브에서 선택 시 title 자동 채움 (`Hotel Check-in` / `Hotel Check-out` / `Hotel Stay`), 사용자 입력 보존 (빈 또는 알려진 auto-title일 때만 덮어씀)
- Trip Services chip 노출에서는 호텔 제외 (`allTripServices.filter(... && !cp.isHotel)`) — 중복 노출 방지
- ⚠️ 호텔 dropdown 정렬: `regularProducts = pickerProducts.filter(cp => !cp.isTripService || cp.isHotel)` — hotel은 trip service 그룹이지만 picker에 노출되어야 하므로 예외 처리

#### 3-1. `applyVariantPick`에서 hotelCheckType 정리
- variant 해제 → `hotelCheckType: null` 명시 클리어 (호텔 product 없을 땐 의미 없음)
- 비-호텔 product로 변경 → `hotelCheckType: null` 클리어 (`cp.isHotel ? {} : { hotelCheckType: null }`)
- auto-title 인식 패턴에 `['Hotel Check-in', 'Hotel Check-out', 'Hotel Stay']` 추가 → 호텔 product 해제 시 자동 채워졌던 hotel 타이틀도 함께 클리어됨

### 4. Fill Transfers (`⇄ Fill Transfers` 버튼)

Day 헤더와 sticky bottom bar 양쪽에 노출. 누르면 해당 day의 인접 항목 쌍 사이에 transfer 자동 생성.

#### 알고리즘
1. day items 정렬 (`compareScheduleItems`), marked-for-removal 제외, **pending 포함**
2. 인접 쌍 (a, b) 순회 — 둘 중 transfer/free/prayer면 skip
3. 그룹 처리:
   - 둘 다 shared(null) → transfer도 shared
   - 한쪽만 shared → 다른 쪽 그룹 사용
   - 둘 다 그룹 → **교집합** 사용, 교집합 없으면 skip
4. 자동 채움:
   - `title = 'Transfer to {b의 location/restaurantName/title}'`
   - `fromLocation = a.location ?? a.restaurantName ?? null`
   - `toLocation = b.location ?? b.restaurantName ?? null`
   - `transportMode = 'car'` (기본값)
   - `time = a.endTime ?? a.time ?? null` (a 이후로 자연 정렬)
5. **정렬 위치 보정**: 새 transfer 삽입 후 day 전체 sortOrder를 0..N으로 재번호 → 시각 순서 보장

#### Idempotent
pending transfer도 pair detection에 포함되므로 두 번 눌러도 같은 자리에 중복 생성 안 됨.

#### 알려진 한계 / 추가 검증 필요
- **(a no-time, b with-time)** 케이스에서 transfer가 b 뒤로 밀릴 가능성 (compareScheduleItems가 timed 항목 우선) — 흔치 않으나 잔존
- 그 외 시뮬레이션에서 의도와 다른 결과 관찰됨 — `notes/PROGRESS.md` 다음 할 일에 재검증 항목 등록

### 6. Admin-side 버그 스윕 (`fix(schedule-editor): admin-side bug sweep`)

서브에이전트 코드 점검 후 발견한 실제 버그 6건 (UX 폴리시 제외).

#### 6-1. `fillTransfers` — pending 항목 sortOrder 충돌
- 기존: `baseItems`가 pending/marked 모두 제외 → 신규 transfer 재번호는 committed만 0..N-1, pending은 기존 sortOrder 유지 → 충돌
- 수정: `baseItems`에 pending 포함(marked-for-removal만 제외), 재번호 pass도 동일 범위 → day 전체 sortOrder 일관성 회복

#### 6-2. `fillTransfers` 중복 생성 (idempotent 아님)
- 기존: 첫 호출 후 transfer가 pending인데 baseItems에서 제외돼 a, b 다시 인접한 것처럼 보임 → 두 번째 호출 시 같은 자리에 transfer 중복 추가
- 수정: §6-1과 같이 baseItems에 pending 포함 → pair detection에서 transfer 마주치면 skip

#### 6-3. `applyVariantPick` — hotelCheckType / Hotel 자동 타이틀 잔존
- §3-1과 같이 처리

#### 6-4. `moveItem` peer 정렬 불일치
- 기존: peers를 `(a, b) => a.sortOrder - b.sortOrder`로 정렬했는데 display는 `compareScheduleItems` (block → time → sortOrder) → ▲/▼ swap이 시각상 인접 행이 아닌 엉뚱한 행과 교환
- 수정: peers도 `compareScheduleItems`로 정렬
- 부수 효과: 시간이 다른 두 항목은 sortOrder 스왑이 시각상 무효(time이 dominant) — 의도된 동작. 시간 같은 항목끼리만 ▲/▼이 실효성 있음

#### 6-5. `commitPendingItem` stale state
- 기존: `saveDraft(items)`의 `items`가 closure 값 — 같은 handler에서 onChange → commit 순으로 일어나면 직전 onUpdate setState 반영 전 상태일 수 있음
- 수정: `const itemsRef = useRef(items); itemsRef.current = items` 매 렌더 → `saveDraft(itemsRef.current)`로 최신 보장

#### 6-6. add 함수들의 sortOrder 충돌
- 기존: `addItem`/`addPrayerTime`/`addFreeTime`/`createTransferFromSelection`/`createHotelBookendFromSelection` 5곳이 `items.filter(i => i.day === day).length`로 sortOrder 결정 → marked-for-removal 포함된 count + sortOrder gap 있으면 collision 가능
- 수정: `nextSortOrderForDay(day)` helper 도입: `Math.max(...sortOrders) + 1`, marked-for-removal 제외. 5곳 전부 helper 사용으로 통일
