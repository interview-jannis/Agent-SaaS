### 3. 상품 멀티 서브카테고리 (Multi-tag) 시스템

**배경:** K-Beauty 상품이 "Stem Cell, Lifting" 같이 복수의 시술 타입에 동시 속해야 함.

#### 3-1. DB 스키마

```sql
CREATE TABLE product_subcategory_tags (
  product_id UUID NOT NULL REFERENCES products(id) ON DELETE CASCADE,
  subcategory_id UUID NOT NULL REFERENCES product_subcategories(id) ON DELETE CASCADE,
  PRIMARY KEY (product_id, subcategory_id)
);
ALTER TABLE product_subcategory_tags DISABLE ROW LEVEL SECURITY;
GRANT ALL ON product_subcategory_tags TO anon, authenticated, service_role;
```

#### 3-2. Excel Upload API (`upload-excel/route.ts`)

```ts
// subcategory 컬럼: "Stem Cell, Lifting" → ["Stem Cell", "Lifting"]
const rawSub = normStr(r.subcategory) ?? ''
const subs = rawSub ? rawSub.split(',').map(s => s.trim()).filter(Boolean) : []

// product upsert 후 tag 동기화
await supabase.from('product_subcategory_tags').delete().eq('product_id', productId)
if (realSubIds.length > 0) {
  await supabase.from('product_subcategory_tags').insert(
    realSubIds.map(sid => ({ product_id: productId, subcategory_id: sid }))
  )
}
```

- `subs[0]`을 `products.subcategory_id` (primary, 하위 호환)로 저장
- 전체 subs를 `product_subcategory_tags`에 동기화

#### 3-3. Agent/Admin UI

`productTagNames(p)` 헬퍼 — tags 배열 읽고, 없으면 primary `subcategory_id` fallback:
```ts
function productTagNames(p: Product): string[] {
  const tags = (p.product_subcategory_tags ?? [])
    .map(t => t.product_subcategories?.name)
    .filter((n): n is string => !!n)
  if (tags.length > 0) return tags
  const primary = p.product_subcategories?.name
  return primary ? [primary] : []
}
```

- `availableSubcategories`: 각 product의 tags 전부 순회
- `productsBySubcategory`: 멀티태그 product가 여러 subcategory 버킷에 동시 노출
- `filteredProducts`: `productTagNames(p).includes(selectedSubcategoryName)`

### 4. PGRST201 — FK 모호성 버그 해결

**문제:** `product_subcategory_tags` 추가 후 상품 페이지에서 "No products found". 에러 로그 없음.

**원인:** `product_subcategory_tags.subcategory_id → product_subcategories` FK가 생기면서 `products → product_subcategories` 경로가 2개(직접 FK + junction 경유)가 돼 PostgREST가 `PGRST201` 에러를 던지지만, 코드에서 `{ data }` 만 구조분해해 에러가 묻힘 → `data: null` → `[]` → "No products found".

**진단 방법:**
1. 에러 로그 추가: `if (productsRes.error) console.error(productsRes.error)` → 콘솔에서 PGRST201 확인
2. 단계별 join 테스트로 `product_subcategories` join이 범인임을 특정

**수정 — FK 이름 명시:**
```ts
// Before
'product_subcategories(name), product_subcategory_tags(product_subcategories(name))'

// After
'product_subcategories!products_subcategory_id_fkey(name), product_subcategory_tags(product_subcategories!product_subcategory_tags_subcategory_id_fkey(name))'
```

- `admin/products/page.tsx`, `agent/product/page.tsx` 양쪽 동일하게 적용

**주의:** 새 테이블 생성 시 다른 테이블과 FK가 겹치면 동일 문제 발생. PostgREST는 ambiguous FK를 에러(null data)로 처리하며 client-side에서는 에러가 묻히기 쉬움. 항상 error도 함께 체크할 것.

### 5. K-Beauty 3단계 카탈로그 (Agent + Admin)

#### 5-1. 구조 결정

K-Beauty 데이터 최종 구조:
- **Row 2 (subcategory):** 시술 타입 — Stem Cell / Lifting / Skin Care (조합 가능)
- **Row 3 (partner pills):** 클리닉 이름 — A/B/C/D/E VIP Premium Dermatology Clinic

#### 5-2. 코드 — K-Beauty partner pills 항상 노출

```ts
// availablePartners / availablePartnerNames 공통 로직
const selectedCatName = categories.find(c => c.id === categoryFilter)?.name ?? ''
const isKBeauty = selectedCatName === 'K-Beauty'
if (!PARTNER_GROUPED_SUBCATEGORIES.has(subcategoryFilter) && !isKBeauty && !allPartnerGrouped) return []
```

K-Beauty 선택 시 subcategory 선택 여부와 관계없이 partner pills 항상 Row 3에 표시.

#### 5-3. Admin Jump-to 카운트 — 전체 기준 고정

```ts
// Before: groupedByCategory.get(cat.id)?.length (필터 적용 후 숫자)
```

### 6. K-Medical Male/Female 별도 상품 유지 결정

**상황:** B/C/D Health Screening Center의 Male/Female description이 사실상 동일 (이사님이 동일하게 입력). Variant로 묶을지 논의.

**결정: 별도 product 유지.**

이유:
1. 실제 검사 항목이 성별별로 달라야 하는 의료 상품 — description 동일한 건 입력 누락, 나중에 분리될 것
2. 운영 시나리오: "Group 1 = 남성 3명, Group 2 = 여성 2명" → 그룹 단위 성별 분리가 자연스러움
3. Junior는 description이 다르므로 어차피 별도 → 한 클리닉에 2~3개 product 불가피
4. Health Screening 파싱 특칙(`variant_label` → 이름 합치기) 그대로 유지

### 7. Bulk 이미지 — 다중 상품 동시 적용 (`bulk-images/route.ts`)

**기존:** `P-001.jpg` → P-001 한 상품에만 적용.

**추가:** 콤마로 여러 상품 번호 지정 가능.

```
P-101,102,103.jpg      → P-101, P-102, P-103에 동시 primary 이미지
P-101, 102, 103-2.jpg  → 공백 허용, 세 상품 모두 2번째 이미지
```

**정규식 변경:**
```ts
// Before: /^p-(\d+)(?:-(\d+))?\.([a-z0-9]+)$/i
// After:  /^p-([\d,\s]+?)(?:-(\d+))?\.([a-z0-9]+)$/i
```

`nums` 배열로 파싱 후 각 product별 독립 처리 (기존 이미지 삭제 → 새 업로드).

### 8. K-Beauty/K-Medical 카드 양식 통일 (`agent/product/page.tsx`)

**문제:** K-Medical은 섹션 헤더 + 카드(flat), K-Beauty는 흰 박스(`bg-white rounded-2xl border shadow`)로 감싼 섹션 카드 — 동일한 3단계 구조인데 시각 양식이 달랐음.

**변경 1 — K-Beauty All 뷰 흰 박스 제거:**
```tsx
// Before
<section className="bg-white rounded-2xl border border-gray-100 shadow-sm p-4">
  <div className="... border-b border-gray-100">

// After
<div>
  <div className="... border-b border-gray-200">  // K-Medical과 동일
```

**변경 2 — K-Beauty subcategory 선택 시 파트너 헤더 그룹핑:**

기존엔 K-Beauty > Stem Cell 선택 시 plain grid (파트너 구분 없음). K-Medical처럼 클리닉별 헤더로 그룹핑되도록 파트너 그룹핑 조건 확장:

```ts
const isKBeautyWithSub = selectedCatName === 'K-Beauty' && selectedSubcategoryName !== ''
return PARTNER_GROUPED_SUBCATEGORIES.has(selectedSubcategoryName) ||
  isKBeautyWithSub ||
  (selectedSubcategoryName === '' && ...)
```

**결과:** K-Beauty All → 시술 타입 섹션(헤더+카드), K-Beauty > Stem Cell → 클리닉 헤더+카드. K-Medical과 완전 동일 구조.
