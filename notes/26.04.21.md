- 해결: `cases.created_at`(스키마상 존재) + `quote.payment_due_date`(이미 저장된 값)로 대체

→ **DB 마이그레이션 불필요**. 기존 quote 5개 모두 정확한 날짜로 표시됨.

### 2. `quote_groups.member_count` 컬럼 추가

**문제**: 인보이스의 Qty가 1로 고정돼 보였음. 이유 — `quote_items.final_price`는 저장 시 `toKRW(p) × memberCount × 마진`으로 이미 총액이 들어가지만, 화면에선 `quote_group_members` row 개수로 memberCount를 역산하고 있었는데 review 페이지에서 사용자가 그룹 배정을 안 하면 이 row가 비어서 fallback 1로 떨어짐.

**해결**:
- SQL: `ALTER TABLE quote_groups ADD COLUMN member_count INT NOT NULL DEFAULT 1;`
- `review/page.tsx` — quote_groups insert 시 `member_count: group.memberCount` 같이 저장
- `quote/[slug]/page.tsx`, `admin/cases/page.tsx`, `agent/cases/page.tsx`, `agent/cases/[id]/page.tsx` — Qty 소스를 모두 `quote_groups.member_count`로 통일

**리스트 테이블의 Members 컬럼**도 `case_members` 개수 → `quote_groups.member_count` 합계로 교체.

### 3. 가격 표시 전 화면 정리

**Agent Home/Review**:
- 카드·카트 총액에 **회사 마진 + 에이전트 마진** 모두 적용
- `toUSD(p) = base × (1 + companyMargin) × (1 + agentMargin) ÷ 환율`
- Home 표시가와 최종 견적가가 비로소 일치

**Admin Products 리스트**:
- 원가(크고 검정) + 에이전트 3 tier(15%/20%/25%) 최종가 병기
- 원화 상품은 원가 아래에 `≈ $환산가` 표시, 3 tier는 **전부 USD**로 통일
- Price 셀 = flex 2열 (원가 우측정렬 / 3 tier 좌측정렬) + 중앙 구분선

**ProductForm 편집 화면**:
- 가격 입력 박스 아래에 "Customer Price by Agent Tier" 프리뷰 박스 (3 tier 실시간 계산)

**USD 2자리 소수점 통일**:
- 모든 USD 출력에 `{ minimumFractionDigits: 2, maximumFractionDigits: 2 }`
- 7개 파일(home, review, cases, cases/[id], clients/[id], admin/products, ProductForm)
- Admin(소수점 2자리)과 Agent(반올림 정수)가 달랐던 불일치 제거

### 4. Admin Cases 상세 — 숨은 3단 버그

증상: 상세 페이지에서 Agent 이름이 "—"로만 뜸. SQL로 직접 JOIN하면 멀쩡함.

원인 추적:
1. **FK 관계 힌트 누락**: Supabase nested select가 `cases.agent_id → agents.id` 관계를 자동 인식 못 함 → `agents(...)`를 `agents!cases_agent_id_fkey(...)`로 명시
2. **RLS 재활성화**: `agents` 테이블 RLS가 어느 시점에 다시 켜진 상태 → `ALTER TABLE agents DISABLE ROW LEVEL SECURITY`
3. **RLS 정책 잔존**: RLS disable 했어도 `authenticated users can read agents` 정책(`auth.uid() = auth_user_id`)이 남아서 PostgREST가 적용 → admin은 자기 auth_user_id와 안 맞는 agent1 row를 못 봄 → `DROP POLICY`로 완전 제거

PROGRESS.md에 "RLS 전체 비활성화" 원칙은 있지만 실제로는 `agents/quote_group_members/schedules`에 정책/rowsecurity가 슬쩍 남아있었음. 이번에 한꺼번에 정리.

**추가 발견**: Invoice 페이지는 같은 코드인데 작동한 이유 → 그 페이지는 `createServerClient()`의 **service role key**로 조회해서 RLS 자체를 우회했기 때문. Admin 페이지는 브라우저 anon 클라이언트라 RLS 직통.

### 6. 무슬림 친화 데이터 구조 확장

**질문 방식 변경**: "Muslim Friendly (Required / Not required)" → **"Muslim? (Yes / No)"**
- 필드는 그대로 `needs_muslim_friendly` (DB 변경 없음), UI 문구만 간결하게
- 대부분 비무슬림 고객이라 "Not required" 같은 부정형 기본값이 어색했음

**DB 확장 — 기도 관련 필드 2개 신규**:
```sql
prayer_frequency TEXT CHECK (... IN ('strict', 'moderate', 'flexible'))
prayer_location TEXT CHECK (... IN ('prayer_room', 'mosque_nearby', 'quiet_private_space', 'any_clean_space', 'no_preference'))
```

**dietary_type ENUM → TEXT+CHECK 마이그레이션**:
- 원래 PostgreSQL ENUM이라 값 추가/삭제/이름변경이 `ALTER TYPE ADD VALUE` → 재생성 dance 필요
- 앞으로 옵션 수정 쉽게 하려고 TEXT + CHECK constraint로 변환
- `products.dietary_type`도 같이 마이그레이션 (같은 enum 참조 중)
- 트러블: `DROP TYPE dietary_type`이 컬럼 DEFAULT 참조 때문에 실패 → DROP DEFAULT → ALTER TYPE → SET DEFAULT → DROP TYPE 순서로 해결

**조건부 폼 UI**:
- Muslim=Yes 선택 시에만 연한 녹색 박스로 Dietary / Prayer Frequency / Prayer Location 3개 드롭다운 노출
- Muslim=No로 토글 시 관련 필드 자동 null 초기화 (저장 시에도 재검증)
- Home 모달 / Clients 모달 / Clients 상세 편집 3곳 모두 동일 적용

- Price 컬럼 셀이 flex 140px+160px 구조라 `<th>`의 기본 위치가 중앙에 떠버렸음
- `<th>`도 같은 flex 구조로 맞춰서 "Price" 라벨이 원가 컬럼 바로 위에 오도록 + `pl-3` 여백

**카테고리별 그룹핑 + Sticky 네비게이션**:
- 상품이 카테고리 섞여있으면 스캔 어려움 → 테이블 안에 카테고리 헤더 row(colSpan=9) 삽입
- 상단 필터 바 아래에 **"Jump to: [Medical(N)] [Beauty(N)] ..."** pill 네비 추가, `sticky top-0 z-20`으로 스크롤해도 항상 보임
- 섹션 헤더는 `sticky top-16 z-10`으로 pill 바 바로 아래에 고정
- pill 하나가 상단으로 스크롤 돌아가는 앵커 역할도 겸함

**Sticky 디버깅 이슈**:
- 처음엔 sticky가 전혀 안 먹힘. admin/layout.tsx의 outer div가 `min-h-screen`이라 main이 내부 스크롤을 안 하고 윈도우 스크롤로 밀려남 → sticky는 "main 기준"이라 함께 사라짐
- 해결: `min-h-screen` → `h-screen` (정확히 100vh 고정) → main이 `overflow-y-auto`로 내부 스크롤 활성 → sticky 정상 동작

### 9. Agent Home 대개편 — 2×2 카테고리 섹션 뷰

**문제 의식**: 카테고리 필터 해제 상태에서 모든 상품이 3열 grid로 한꺼번에 쏟아짐 → 카테고리 인지도 낮고 많아지면 스크롤만 길어짐.

**최종 구조**:
- 필터 해제: **2×2 카드 컨테이너** (Medical / Beauty / Wellness / Subpackage 각 quadrant)
  - 각 섹션에 3개 프리뷰 + "See all (N) →" 버튼
  - `grid-cols-2 grid-rows-2 h-full` + `overflow-hidden`으로 **세로 스크롤 없음** 보장
  - 섹션 내부 카드는 compact 모드 (이미지 aspect-video, description 숨김)
- 카테고리 단일 선택: 해당 카테고리만 **flat grid-cols-4**로 전체 노출

**사이드바 카테고리 필터**: 다중 선택 체크박스 → **단일 선택 라디오** + "All" 옵션. 섹션 토글 ↔ flat grid 토글이 한 state에서 통제됨.

**장바구니 pinned 정렬**:
- 선택한 상품이 한참 아래에 있으면 필터 해제했을 때 프리뷰 상단 3개에 안 보일 수 있음
- 섹션 뷰 "진입 시점"에 카트 상품 ID 스냅샷을 잡아 해당 상품을 섹션 앞쪽으로 정렬
- 카트에 담는 순간마다 재정렬하면 카드가 휙휙 움직여서 어지러움 → 스냅샷 방식이 차분함

**Subpackage 카테고리 신설**: `INSERT INTO product_categories (name, sort_order) VALUES ('Subpackage', 4)`.

## 느낀 점 / 메모

- **PostgreSQL ENUM은 MVP에 비효율**: dietary처럼 값이 유연해야 하는 옵션은 처음부터 `TEXT + CHECK`로 가는 게 낫다. ENUM의 장점(타입 안전성)은 CHECK로도 얻어지고, 값 추가/삭제/이름변경은 CHECK가 압도적으로 쉬움. 레슨 학습 완료.
- **"RLS는 비활성화 상태"라는 기억은 믿으면 안 됨**: 언제 어느 테이블이 다시 켜져있을지 모름. Admin 쪽 이슈는 정확히 이 가정이 흔들려서 몇 시간 잡아먹음. PROGRESS.md에 기록된 내용이 DB 실제 상태와 일치하는지 가끔 재검증 필요.
- **Supabase PostgREST FK 인식은 까다로움**: 같은 `cases.agent_id → agents.id` 관계인데 Invoice(server + service role)에서는 작동, Admin(browser + anon)에서는 null. FK hint(`!constraint_name`)를 명시하면 일관적. 앞으로 nested select에선 명시적으로 쓰는 게 안전.
- **가격 계산은 SSOT(Single Source Of Truth) 원칙이 중요**: Home·Review·Cases 상세·Invoice가 모두 같은 공식(`base × (1+company) × (1+agent) / xr`)을 쓰게 통일한 뒤부터 버그가 딱 끊김. 여기저기서 재구현하면 반드시 어딘가 어긋남.
- **미팅 피드백은 바로 적용하는 게 편함**: 4/17 미팅 피드백(Tiktak 브랜딩, 카테고리 순서, Add Client 분리)이 며칠 지나 구현되니 UX 맥락이 흐려짐. 다음 미팅부터는 1~2일 내 처리.
- **레이아웃 하나 바꾸는 게 나비효과**: `admin/layout.tsx`의 `min-h-screen → h-screen` 한 글자 차이로 sticky가 살아났음. CSS 레이아웃은 작은 속성이 큰 행동 차이를 만든다는 걸 또 확인.
