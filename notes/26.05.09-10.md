## 5. Trip Services 분리 — 상품 선택 플로우 전면 개편

### 배경

Subpackage 상품(통역, 차량, 호텔, 컨시어지)도 기존 그룹(그룹 A, B, Shared)에 넣는 구조였는데,
그룹 단위는 **인원 수** 기준인 반면 Subpackage는 **일 수** 기준이어야 해서 맞지 않음.
통역 3일, 차량 5일을 표현하려면 별도 그룹을 생성해야 해서 그룹이 급증하는 문제.

### 해결책

| 카테고리 | 위치 | 단위 |
|---|---|---|
| K-Medical / K-Beauty / K-Wellness 등 | 기존 그룹 (A, B, Shared) | 인원 수 × 단가 |
| Subpackage (통역, 차량, 호텔, 컨시어지) | **Trip Services** 별도 섹션 | 일 수 × 단가 |

### 핵심 개념

- `TripServiceItem = { productId, variantId, days }` 신규 타입
- `CART_VERSION = 3` — 구형 cart localStorage 자동 무효화
- `isSubpackageProduct(p)` — `p.product_categories?.name === 'Subpackage'`
- `toggleServiceItem()` — Trip Services 추가/제거
- `setServiceDays()` — 일 수 +/- 조절
- 호텔 상품: `days` 기본값 = `nightsBetween(dateStart, dateEnd)` (자동)
- **Shared 그룹** 이름 변경 불가 고정 (double-click 비활성화, 연필 아이콘 숨김)

### DB 마이그레이션 없이 name-based convention으로 처리

`document_groups`에 `group_type` 컬럼 추가 시도했으나:
- Supabase JS 클라이언트에 `exec_sql` RPC 없음
- Management API는 personal access token 필요 (service role key 불가)

→ DB 변경 없이 이름으로 타입 파생:

```typescript
// src/lib/documents.ts
export type GroupType = 'regular' | 'shared' | 'trip_services'

export function groupTypeFromName(name: string | null): GroupType {
  if (name === 'Shared') return 'shared'
  if (name === 'Trip Services') return 'trip_services'
  return 'regular'
}
```

## 6. 견적서 불변성(Immutability) 모델 + Finalize Pricing 리팩터

### 배경 및 설계

기존 Edit Selected Products는 `quotation` document_items를 직접 수정. 문제:
- 견적서(Quotation)은 에이전트가 보낸 snapshot — 고객이 link 열어 볼 수 있는 문서.
- Admin이 항목을 편집하면 고객이 보는 견적서가 소급하여 바뀌는 사태 발생.
- 정확한 audit trail 없음 (original vs admin_added 구분 어려움).

### 해결책 — Dual Document 모델

| 문서 | 역할 | 편집 가능 |
|---|---|---|
| `quotation` | Agent 발행 snapshot — 불변 | ❌ 절대 수정 금지 |
| `final_invoice` (draft) | Admin 편집 작업본 — 초안 | ✅ finalize 전까지 |

- Agent가 견적서 발행 → `quotation` row 생성.
- Admin이 Edit Selected Products 클릭 → `final_invoice` draft가 없으면 **lazy 생성** (`createDraftFinalInvoice`).
  - `createDraftFinalInvoice`는 내부적으로 `issueInvoice({ copyItemsFromQuotation: true })` 호출 → quotation items를 final_invoice로 복사.
- 이후 모든 item add/remove → `final_invoice` document_items 대상.
- `quotation` document_items는 건드리지 않음.

### 구현 상세

**`src/app/admin/cases/[id]/page.tsx`**

- `editGroups` = `finalInvoice.document_groups` (편집 타겟, `final_invoice` 기준)
- `sortedGroups` = `latestQuote.document_groups` (quotation, 표시 전용)
- Edit Selected Products 섹션:
  - 렌더: `sortedGroups.map(...)` → `editGroups.map(...)`
  - Picker 기본 group: `sortedGroups[0].id` → `editGroups[0]?.id`
  - 조건: `!latestQuote.finalized_at` → `!finalInvoice?.finalized_at`
  - `canEditProducts = canEdit && !!finalInvoice && !finalInvoice.finalized_at`
  - 열 때 `finalInvoice` 없으면 `createDraftFinalInvoice(caseId)` 호출 → spinner 표시
- Finalize Pricing 섹션:
  - 조건: `finalInvoice &&` (기존엔 quotation 기반)
  - 모든 items/totals: `editGroups` 기반 (final_invoice items)
  - 첫 finalize: `finalizeDocument(finalInvoice.id)` (document_number 발번 + finalized_at 세팅)
  - 재 finalize(reprice): `repriceDocument(finalInvoice.id, ...)` 호출
  - `issueInvoice` / `syncFinalInvoiceFromQuotation` / `getCaseFinalInvoice` import 전부 제거
- Financials 섹션:
  - `isFinalized = !!finalInvoice?.finalized_at`
  - `displayTotal = isFinalized ? finalInvoice.total_price : latestQuote.total_price`
  - Pending changes → `finalInvoice.document_groups` 기반
  - Payment Due → `finalInvoice.payment_due_date`

### DB 트리거 — quotation items 변경 원천 차단

**`supabase/migrations/quotation_items_immutable.sql`**

```sql
CREATE OR REPLACE FUNCTION prevent_quotation_item_mutation()
RETURNS trigger LANGUAGE plpgsql AS $$
DECLARE doc_type TEXT;
BEGIN
  SELECT d.type INTO doc_type
  FROM documents d
  WHERE d.id = COALESCE(NEW.document_id, OLD.document_id);

  IF doc_type = 'quotation' THEN
    RAISE EXCEPTION 'document_items for quotation documents are immutable. Target the draft final_invoice instead.';
  END IF;

  RETURN COALESCE(NEW, OLD);
END;
$$;

DROP TRIGGER IF EXISTS trg_quotation_items_immutable ON document_items;
CREATE TRIGGER trg_quotation_items_immutable
  BEFORE INSERT OR UPDATE OR DELETE ON document_items
  FOR EACH ROW EXECUTE FUNCTION prevent_quotation_item_mutation();
```

애플리케이션 레이어 뿐 아니라 DB 레벨에서도 quotation 항목 변경을 막아 2중 보호. 트리거는 Supabase SQL 에디터에서 직접 실행 필요 (migration 파일은 작성 완료).

## 8. Guide 콘텐츠 전면 재작성

### 배경

기존 가이드는 "무엇을 할 수 있다"만 나열. 시뮬레이션 앞두고 현업 사용자가 **왜** 그 단계에서 그 액션을 해야 하는지 이해할 수 있도록 전면 재작성.

### 변경 사항

#### CASE_STEPS (케이스 진행 흐름 아코디언) — 양쪽 가이드 공통

10개 status 단계 각각에 대해:
- **Admin 관점** / **Agent 관점** / **Client 관점** 분리.
- 각 관점별 단락(`\n\n` 구분) — "왜 이 단계인가", "무엇을 해야 하나", "다음은 무엇인가" 흐름.
- 특히 강조한 내용:
  - `awaiting_contract`: Quotation = snapshot, 이후 수정 불가. 고객은 `/quote/[slug]`로 확인.
