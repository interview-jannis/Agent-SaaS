## 2. 인당 견적서 — `?member=` 탭 방식

### 구현

**`src/components/QuoteDocument.tsx`**

- `filterMemberId?: string` prop 추가.
- Supabase 쿼리: `cases > case_members(id, is_lead, clients(name, ...))` + `document_groups > document_group_members(case_member_id)` 포함.
- 멤버 2명 이상 시 문서 상단에 sticky 탭 바 렌더:
  - "All Members" (기본) + 각 멤버 이름 탭 (★ = lead)
  - 탭 클릭 → `?member={case_member_id}` 쿼리로 페이지 재렌더
  - `print:hidden` — 인쇄 시 탭 바 숨김
- 멤버 필터 시 그룹 필터링: 해당 멤버가 속한 그룹만 표시 (`document_group_members.case_member_id` 기준)

**`src/app/quote/[slug]/page.tsx`** / **`src/app/invoice/[slug]/page.tsx`**

- `searchParams.member` 읽어서 `filterMemberId`로 전달. quotation/invoice 두 라우트 모두 적용.

### 결정 사항

- 서브루트(`/member/[memberId]`) 방식 폐기 → 단일 URL + 쿼리 파라미터 방식 채택.

## 3. Admin 권한 체계 재설계

### 배경 및 설계 논의

기존에는 모든 admin이 모든 케이스 작업(스케줄 업로드, finalize pricing, Mark Paid 등) 가능.
5/9 논의 결과:

- **Onboarding 계약서 counter-sign**: super admin만 (법적 서명 — 대표님)
- **Approve & Activate**: super admin만 + Approve 시 담당 admin 지정
- **케이스 작업**: 담당 admin(assigned_admin_id) 또는 super admin만

Super admin은 복수 계정 가능 (`is_super_admin` boolean). 나중에 대표님 계정 생기면 super admin으로 추가 지정.

### 구현

#### 1) Onboarding counter-sign — super admin 전용

**`src/app/api/admin/sign-contract/route.ts`**

- admin fetch 시 `is_super_admin` 컬럼 추가.
- `!adm.is_super_admin` → 403 반환.

**`src/app/admin/agents/[id]/contract/[contractId]/page.tsx`**

- `adminProfile`에 `is_super_admin` 포함.
- `adminProfile.is_super_admin`이 false면 사인 버튼 대신 amber 안내 메시지 표시.

#### 2) Approve & Activate — super admin 전용 + 담당 admin 지정 모달

**`src/app/admin/agents/[id]/page.tsx`**

- `isSuperAdmin` 상태 기존에 있었으나 버튼에 미적용 → 적용.
- `!isSuperAdmin`인 admin에겐 버튼 대신 amber 안내 메시지.
- Approve 버튼 → 모달 트리거로 변경 (`window.confirm` 제거).
- 모달에 admin 드롭다운 (★ = super admin 표시) → 담당 admin 선택.
- 확인 시 `assigned_admin_id` = 선택한 admin으로 설정.
- `approveAgent()` 함수: `assignToAdminId` 파라미터 반영.

#### 3) 케이스 작업 canEdit 게이트

**`src/app/admin/cases/[id]/page.tsx`**

- `init()` 내 현재 admin fetch (`auth_user_id` → `id, is_super_admin`).
- `agents` join에 `assigned_admin_id` 추가.
- `canEdit = isSuperAdmin || agent.assigned_admin_id === currentAdminId`
- `!canEdit` 시 케이스 상단 amber "View only" 배너 표시.

**적용 범위 (6곳)**:

## 5. Trip Services 분리 — 상품 선택 플로우 전면 개편 (컨텍스트 이어 작업)

### 배경

Subpackage 상품(통역, 차량, 호텔, 컨시어지)도 기존 그룹(그룹 A, B, Shared)에 넣는 구조였는데,
그룹 단위는 **인원 수** 기준인 반면 Subpackage는 **일 수** 기준이어야 해서 맞지 않음.
통역 3일, 차량 5일을 표현하려면 별도 그룹을 생성해야 해서 그룹이 급증하는 문제.

### 해결책

| 카테고리 | 위치 | 단위 |
|---|---|---|
| K-Medical / K-Beauty / K-Wellness 등 | 기존 그룹 (A, B, Shared) | 인원 수 × 단가 |
| Subpackage (통역, 차량, 호텔, 컨시어지) | **Trip Services** 별도 섹션 | 일 수 × 단가 |

### 핵심 개념

- `TripServiceItem = { productId, variantId, days }` 신규 타입
- `CART_VERSION = 3` — 구형 cart localStorage 자동 무효화
- `isSubpackageProduct(p)` — `p.product_categories?.name === 'Subpackage'`
- `toggleServiceItem()` — Trip Services 추가/제거
- `setServiceDays()` — 일 수 +/- 조절
- 호텔 상품: `days` 기본값 = `nightsBetween(dateStart, dateEnd)` (자동)
- **Shared 그룹** 이름 변경 불가 고정 (double-click 비활성화, 연필 아이콘 숨김)

### DB 마이그레이션 없이 name-based convention으로 처리

`document_groups`에 `group_type` 컬럼 추가 시도했으나:
- Supabase JS 클라이언트에 `exec_sql` RPC 없음
- Management API는 personal access token 필요 (service role key 불가)

→ DB 변경 없이 이름으로 타입 파생:

```typescript
// src/lib/documents.ts
export type GroupType = 'regular' | 'shared' | 'trip_services'

export function groupTypeFromName(name: string | null): GroupType {
  if (name === 'Shared') return 'shared'
  if (name === 'Trip Services') return 'trip_services'
  return 'regular'
}
```
