## 1. 핵심 서비스 구조 및 운영 플로우

플랫폼은 에이전트 유입부터 계약, 교육, 실제 영업 수행까지 이어지는
**사전 검증 → 계약 → 가입 → 운영 구조**로 설계.

### ① 에이전트 사전 온보딩 (Pre-Onboarding)
에이전트는 회원가입 이전 단계에서 충분한 정보 확인 및 의사결정을 진행.

- 제공 항목:
  - 사업 소개 및 서비스 구조
  - **오리엔테이션(OT) 자료**
  - NDA 및 파트너십 계약서
- 목적:
  - 에이전트가 서비스 이해 및 영업 방식 숙지
  - 계약 체결 이전 단계에서 **충분한 검토 시간 확보**

### ② 계약 체결 (전자서명 기반)
- OT 및 자료 검토 완료 후 NDA + 파트너십 계약서 → **모바일 기반 전자서명**
- 특징:
  - 종이 계약 제거
  - 간편 계약 프로세스 구축
  - 계약 완료 시점에만 다음 단계 진행 가능

### ③ 에이전트 회원가입 및 계정 생성
- 계약 완료 이후 에이전트 직접 회원가입
- 입력 정보:
  - 기본 프로필
  - **정산 계좌 정보 (필수)**
- 결과: 에이전트 권한 부여, 시스템 접근 가능

### ④ 고객 상담 및 견적 생성
- 에이전트가 수행: 고객 등록 → 상품 선택(그룹/가족 단위 포함) → 견적 생성
- 기능:
  - 인원 기반 자동 가격 계산
  - 환율 반영 (원화/달러)
  - 견적 생성 후 케이스 등록

### ⑤ 결제 및 운영 처리
- 결제 완료 이후 내부에서 일정 등록
- 시스템 자동 처리: 일정 URL 생성 및 에이전트·고객에게 전달

- Admin Products에서 상시 편집
- **DB → Excel Export 버튼**으로 엑셀 형태 확인 언제든 가능 (비기술자도 접근 편의)

## 전자서명 방식 선정 과정

### 자체 구현 (채택)
- HTML canvas에 마우스/터치 서명 → PNG 이미지 → Supabase Storage 저장
- 서명자 정보(이메일, IP, 디바이스, 타임스탬프) DB 기록
- **SMS 본인 인증 병행** (무료 API 활용, "난 서명한 적 없다" 반박 차단)
- **서명 완료 PDF를 에이전트 이메일로 즉시 발송** (사본 보관 증빙)

### 외부 서비스 (v2 검토 대상)
- Modusign(모두싸인) 또는 DocuSign — 법적 효력·감사추적 강함, 월 고정비
- 단점: 발송→수신→서명 비동기 흐름이라 가입 UX 한 템포 끊김 (이메일/SMS 열어야 함) → 초기 에이전트 전환율에 악영향

### 채택 이유
- 카드 단말기 서명 = 기술적으로 canvas 서명과 동일 원리. 국내 스타트업 B2B 계약 다수가 이 수준에서 운영
- NDA/파트너십 수준의 분쟁 리스크는 초기에 제한적
- 자체 구현 + SMS 인증 + PDF 발송 조합이면 실무적으로 카드 서명보다 강한 증빙 확보
- 에이전트 수 증가 또는 분쟁 리스크 상승 시 Modusign으로 마이그레이션 — 기존 서명 데이터 보존 + 신규만 외부 서비스 전환 가능

## 상품 데이터 전략 선정 과정

### 채택: SaaS 직접 등록 + Excel Export

**이유**:
- 현재 상품 수 적음 + Admin Products UI 이미 완성 (이미지 업로드 포함)
- Bulk import 개발 시 "이미지 매칭" 문제 해결 필요 (엑셀 URL? 파일명? 별도 업로드?) → 복잡도 +1일 이상
- 상품은 1회 등록 후 지속 편집 대상 → Admin UI가 실질 source of truth가 돼야 함
- 엑셀과 SaaS를 양방향 동기화하면 "어느 쪽이 맞냐" 분쟁 발생 (실무 최빈 사고)

**Excel Export 구현 방향**:
- Admin Products 페이지 우상단에 "Export to Excel" 버튼
- `xlsx` 라이브러리로 .xlsx 파일 다운로드
- 카테고리명·환산가격 등 가공 형태로 export (raw DB보다 읽기 편함)

**재검토 기준**: 상품 수 200개 이상으로 증가 시 Bulk Import UI 개발 (UPSERT + 미리보기 + 이미지 매칭 포함)
