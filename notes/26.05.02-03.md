## 1. Status state machine 재설계 (8 → 11단계)

### 흐름 변경
- 이전: `awaiting_info → awaiting_schedule → reviewing_schedule → awaiting_pricing → awaiting_payment → awaiting_travel → completed`
- 신규 SOP 흐름: `awaiting_contract → awaiting_deposit → awaiting_info → awaiting_schedule → reviewing_schedule → awaiting_pricing → awaiting_payment → awaiting_travel → awaiting_review → completed`

### 신규 status 3개
- `awaiting_contract` — 견적 발송 후, 3자 계약 사인 대기
- `awaiting_deposit` — 계약 체결 후, deposit 정산만 진행 (Client→Agent + Agent→Admin 양 다리)
- `awaiting_review` — 여행 종료 후, 클라이언트 설문 제출 대기

### 자동 전이 정책 (caseTransitions.ts)
- `awaiting_contract → awaiting_deposit`: case_contracts 3자 사인 모두 채워질 때 (`tryAdvanceContractSigned`)
- `awaiting_deposit → awaiting_info`: 두 deposit 다리 모두 paid (info 미완성이면 awaiting_info, 완성이면 awaiting_schedule 직행)
- `awaiting_info → awaiting_schedule`: 정보 완료 (legacy 경로도 재사용)
- `awaiting_review → completed`: 설문 제출 (`tryAdvanceReviewSubmitted`)

### Self-heal
- 케이스 상세 페이지 로드 시 `notifyCaseInfoChanged` 자동 호출 → stuck case 자가 진행
- DB 직접 수정 / 이전 코드로 paid했던 케이스 모두 페이지 한 번 열면 정정

### 흐름 분리 — 사용자 피드백 정정
처음엔 `awaiting_deposit`을 deposit + info 병렬 진행으로 만들었는데, 사용자가 "deposit 정산만 하고 끝나면 awaiting_info로" 명시. 분리:
- awaiting_deposit = deposit 정산 only (양 다리)
- awaiting_info = 정보 입력 only

### Hero 문구 분리
- 이전: "Deposit + client info pending / Trip info ✗ · Clients ✓ · Groups ✗"
- 신규: awaiting_deposit은 deposit-only ("Client → Agent ✓ · Agent → Admin ✗"), awaiting_info는 info-only ("Trip info ✗ · Clients ✓ · Groups ✗")

## 2. 3자 계약서 (case_contracts)

### 스키마
```sql
CREATE TABLE case_contracts (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  case_id UUID NOT NULL REFERENCES cases(id) ON DELETE CASCADE,
  contract_type TEXT CHECK (contract_type IN ('agent_client', 'three_party')),
  title_snapshot TEXT, body_snapshot TEXT,
  client_token TEXT UNIQUE,           -- /case-contract/[token]
  -- Agent / Client / Admin 각각 sig + signed_at + signer_name
  agent_signed_at, agent_signature_data_url, agent_signer_name,
  client_signed_at, client_signature_data_url, client_signer_name,
  admin_signed_at, admin_signature_data_url, admin_signer_id, admin_signer_name, admin_signer_title,
  ip_address, user_agent, created_at,
  UNIQUE (case_id, contract_type)
);
```

### 흐름
1. Agent가 케이스 상세에서 "Generate Contract" → contract_templates에서 `three_party` body 가져와 토큰 치환 후 case_contracts row 생성
2. Agent SignaturePad로 사인
3. 클립보드로 `/case-contract/[token]` URL 복사 → 클라이언트에 전달
4. 클라이언트가 공개 페이지에서 자기 이름 입력 + 사인
5. Admin이 case 상세에서 카운터사인 → 자동 status 전이

### 토큰 치환
`{{AGENT_NAME}}`, `{{AGENT_COUNTRY}}`, `{{CLIENT_NAME}}`, `{{CASE_NUMBER}}`, `{{QUOTE_NUMBER}}`, `{{TOTAL_AMOUNT}}`, `{{DEPOSIT_PERCENTAGE}}` — 생성 시점에 body_snapshot에 치환 후 저장 (audit 무결성).

### 신규 컴포넌트
- `src/lib/caseContracts.ts` — createCaseContract / signAsAgent/Client/Admin / tryAdvanceContractSigned
- `src/components/CaseContractViewer.tsx` — 본문 + 3개 sig slot + 사인 모드별 SignaturePad
- `src/components/AgentCaseContractSection.tsx` — Generate / Agent sign / Copy Client Link
- `src/components/AdminCaseContractSection.tsx` — Counter-sign UI
- `src/app/case-contract/[token]/page.tsx` — Client 공개 사인 페이지

## 8. 알림 신뢰성 회복

### notifyAgent server-first
이전: notifyAgent client-side INSERT — 크로스 유저 INSERT 시 RLS/세션 경계로 silent fail 가능성
신규: `/api/notifications/notify-agent` (service role) 우선 + client fallback

### 알림 트리거 추가 (5종)
- 3자 계약 체결: agent + admin 양쪽 (tryAdvanceContractSigned)
- Agent → Client deposit invoice 발행: admin 알림
- Mark Paid 시 counterparty 알림 (deposit / balance / commission)
- 계약 단계 시작 (`awaiting_deposit`) 진입: agent에게 "deposit 발행하세요"

### CaseDocumentsSection 직접 INSERT 제거
이전: `supabase.from('notifications').insert({ target_type: 'admin', target_id: null, ... })` → auth_user_id 없어서 admin bell이 못 잡음
신규: notifyAllAdmins 사용 (server-side 브로드캐스트)

## 결정 사항

| 항목 | 결정 | 이유 |
|------|------|------|
| Status 흐름 | 11단계 | SOP 1~15 단계 모두 status로 표현 |
| awaiting_deposit 정의 | deposit only (info 분리) | 한 단계에 두 종류 작업 섞이면 UI 헷갈림 |
| Deposit "fully paid" | Client→Agent + Agent→Admin 양 다리 | "정산 완료" = 진짜 admin까지 들어가야 |
| 3자 계약 진행 방식 | per-case row + client_token URL | 케이스마다 별도 사인 필요. agent_contracts와 분리 |
| 설문 질문 관리 | system_settings jsonb + admin UI | 별도 테이블 과도, jsonb 단순함 |
| Stamps 저장 | Storage bucket public + URL 컬럼 | 도장은 invoice에 매번 렌더, 빠른 액세스 |
| Stamp render 방식 | absolute 우상단 + mix-blend-mode multiply | 자연스러운 인감 느낌 |
| Hero 분리 정책 | status별 한 종류 액션만 | "deposit + info" 같이 묶으면 시선 분산 |
| 박스 통합 | Trip Setup / Financials 각각 한 박스 | 사용자 피드백 — 박스 너무 많음 |
| Auto-collapse | 정보 다 차면 자동 접힘 | 완료된 박스가 화면 점유하지 않게 |
| 잠금 정책 (awaiting_contract) | trip/members/deposit 모두 잠금 | 계약 전엔 어떤 행동도 안 됨 |
| notifyAgent 신뢰성 | server-first + client fallback | 4/28 admin broadcast 같은 실패 방지 |
| awaiting_info 의미 | 두 가지 (legacy + 신규 SOP) | 같은 status 재사용, 의미 자연스럽게 동일 |
| Self-heal 위치 | 케이스 detail 페이지 로드 시 | 모든 stuck case 자동 정정 |

## 느낀 점

- **Status 흐름 redesign이 가장 무거웠음**: 11단계로 늘리니 status 체크하는 모든 곳이 영향. agent + admin case detail 양쪽, dashboard, settlement, payouts 등 ~15곳 일괄 수정. caseStatus.ts 단일 소스 패턴이 살려줘서 그나마 빠르게 정리.
- **사용자 피드백 사이클이 가치 큼**: deposit_paid 정의 (한 다리 → 양 다리), Hero 문구 분리, Send Contract 라벨, 박스 통합 — 모두 사용자가 보고 즉시 짚어준 것들. 혼자 결정으로 가면 한 두 번은 잘못 가게 됨.
- **자동 E2E 검증의 효과**: Claude Preview MCP로 한 바퀴 돌리면서 fixCommit-가능한 버그들을 발견 (login form 이벤트, SURVEY_QUESTIONS stale, deposit 양 다리 검사). 손으로 검증했으면 더 오래 걸렸을 부분.
- **Self-heal 패턴**: 페이지 로드 시 자동 advance — DB에 stuck된 케이스 정정에 효과적. 5/1 회고에서 "테스트 없이 push 안 함" 다짐과 결합하면, 실제 사용 중에 발견되는 작은 inconsistency를 자동으로 메우는 안전망.
- **박스 통합 + auto-collapse 조합이 의외로 강력**: 정보가 다 차면 안 보이게 → 시선이 액션 필요한 곳에만 머묾. SaaS UX의 기본인데 처음부터 적용 안 했던 게 아쉬움. case detail이 1900줄짜리 거대 파일이 된 이유 중 하나.
- **데이터 모델 vs UI 모델의 분리**: `awaiting_info` 의미가 두 가지(legacy / 신규 SOP) — 같은 DB 값이지만 흐름 컨텍스트가 다름. 코드는 둘을 구분 안 하고 (정보 모자라면 advance 안 함) 자연스럽게 처리. status 의미가 시간이 지나며 바뀌어도 작동.

## 1. Finalize Pricing 강화 (admin/cases/[id])

### Pre-finalize 항목 추가/삭제
- 기존: `final_price` 인라인 편집만 가능
- 신규: 행 우측 `×` (삭제 마킹 → strikethrough + Undo) + 그룹 하단 검색 콤보박스로 새 상품 추가
- 검색: 이름/파트너 매칭 popover 최대 20개. 같은 그룹 내 중복 차단.
- staged 상태로 보관 → "Finalize Pricing & Issue Invoice" 클릭 시 일괄 반영 (delete → insert → update 순)
- finalize 후엔 가격 편집만 노출 (추가/삭제는 Additional Invoice로 처리)
- audit log details에 `items_added` / `items_removed` 카운트 포함

### Multi-currency 입력 (KRW ↔ USD)
- 데이터 모델 무변경 — 저장은 항상 KRW canonical
- 각 행에 세그먼트 토글 `[₩|$]` (단일 토글 버튼이 아님 — 두 옵션 동시 노출이 어포던스 강함)
- USD 입력 시 `Math.round(usd × exchangeRate)`로 KRW 환산 저장
- 신규 항목은 product의 `price_currency`에 따라 자동 시작 단위 설정 (USD product → $로 시작)
- 기존 항목 토글 시 "orig" 라벨도 같은 단위로 변환 표시

### 결정
- 옵션 B (네이티브 통화 컬럼 추가)는 over-engineering — final_price는 client가 KRW로 얼마 내는지가 본질. USD 입력은 admin 편의 문제.
- product catalog (`products.price_currency`)는 그대로 유지 — 네이티브 통화 메모리는 거기 보존.

## 2. Agent-Admin Assignment 시스템

### 동기
모든 admin에게 모든 알림이 가는 게 노이즈. 카톡/왓츠앱 연동 들어가면 더 심해질 거. 한 케이스/agent를 한 admin이 owner로.

### 디자인 결정
- claim 단위: **agent** (case가 아니라). super admin이 agent 생성 시 담당 admin 고정 → 그 agent의 모든 케이스 자동 owner.
- 자동 채움 시점: **승인** (NDA/Partnership 검토하고 Approve 누른 admin이 그대로 owner). agent_contracts.approved_by와 일관.
- 비-owner 정책: **soft 락** — 알림은 owner에게만, 편집은 누구나 가능 (audit log에 actor 남음). 운영 사고 방지.
- 재할당: super admin만 가능, audit log `agent.reassigned`에 from/to 기록.
- 폴백: `assigned_admin_id`가 NULL이면 super admin들에게 broadcast.

### 스키마
```sql
ALTER TABLE agents
  ADD COLUMN IF NOT EXISTS assigned_admin_id uuid
    REFERENCES admins(id) ON DELETE SET NULL;
CREATE INDEX IF NOT EXISTS idx_agents_assigned_admin_id
  ON agents(assigned_admin_id);
```
- `ON DELETE SET NULL` — admin 삭제 시 agent는 자동으로 super admin 폴백.

### 신규 헬퍼 + API
- `notifyAssignedAdmin({agent_id?, case_id?}, message, link)` — `case_id`만 들고 있어도 자동으로 agent 조회 → assigned_admin 라우팅.
- `/api/notifications/notify-assigned-admin` — service role insert (RLS/세션 경계 우회).
- 폴백: assigned_admin 없으면 super admin들에게 broadcast.

### Phase 2 — 호출부 교체 (11곳)
- `caseTransitions.ts` (5), `surveys.ts` (1), `caseContracts.ts` (1), `CaseDocumentsSection.tsx` (3), `AgentCaseContractSection.tsx` (1), `case-contract/[token]/page.tsx` (1), `agent/home/review/page.tsx` (1), `agent/cases/[id]/page.tsx` (4)
- 남긴 곳: `ContractStep.tsx` (agent 승인 전이라 assigned_admin 없음 → broadcast 유지)

### UI
- Admin Agents list — "Assigned to" 컬럼 추가 (데스크탑만, 미배정은 회색 "Unassigned")
- Admin Agent detail — 상단에 "Assigned to: [name]" 박스 + super admin에게만 보이는 **Reassign** 버튼 (모달 dropdown + audit)

### Backfill
```sql
UPDATE agents
SET assigned_admin_id = (
  SELECT id FROM admins WHERE is_super_admin = true ORDER BY created_at LIMIT 1
)
WHERE onboarding_status = 'approved' AND assigned_admin_id IS NULL;
```

## 느낀 점 (Day 2)

- **사용자 피드백 라운드의 효율성**: 5/2에 SOP 풀로 깔고 5/3에 사용성 라운드 — 이 분리가 좋음. 기능을 만들 때는 흐름에 집중하고, 다듬을 때는 한 번에 모아서. 한꺼번에 하면 "어디까지가 필수냐"가 흐려짐.
- **세그먼트 토글의 발견성**: 단일 ₩↔$ 토글보다 [₩|$] 두 옵션 동시 노출이 어포던스 측면에서 훨씬 나음. 공간도 거의 안 먹음. 다른 곳에도 적용 검토할 만함.
- **agent-admin assignment의 진입 비용 vs 효과**: 데이터 모델 한 컬럼 추가 + 헬퍼 1개 + UI 2곳 + 11곳 호출 교체. 작업량은 크지만 카톡/왓츠앱 연동 들어갈 때 노이즈 폭발 막아주는 기반. 일찍 했어야 했나 싶지만, MVP 흐름이 충분히 안정된 다음 깔아야 polluted call sites가 안 생김.
- **soft 락 정책의 가치**: 처음엔 "비-owner는 view-only"가 자연스러워 보였는데, 운영 현실 (휴가/부재) 생각하면 hard 락은 사고 유발. 알림만 분기하고 권한은 유지 — 가시성·접근성 분리가 핵심.
- **invoice 카드 톤다운**: 정보 위계 (요약 < 액션) — financials 합계가 invoice 카드보다 시각적으로 가벼우면 안 됨. 카드를 정보 위계의 어느 층에 두느냐가 디자인의 본질.
