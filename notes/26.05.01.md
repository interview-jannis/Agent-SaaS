## 2. Documents 모델 — Phase 1 (스키마 + 마이그레이션)

기존 `quotes` 1 row가 Quotation + Invoice 둘 다 역할 → 4/30 SOP의 4가지 invoice 종류 (디파짓/잔금/추가/커미션) 표현 불가능. 통합 `documents` 테이블로 갈아끼움.

### 새 테이블 4개
- `documents` — type 5종 (quotation, deposit_invoice, final_invoice, additional_invoice, commission_invoice), document_number, slug, total_price, finalized_at, payment_due_date, payment_received_at, signer_snapshot, first_opened_at, open_count
- `document_groups` — 문서별 그룹 (가족 단위 등)
- `document_items` — 문서별 라인 아이템
- `document_group_members` — 그룹 ↔ case_member 매핑

### 마이그레이션 SQL
- `quotes` → `documents(type='quotation')` UUID 보존 (schedules.quote_id 그대로 작동)
- `finalized_at IS NOT NULL` 인 quotes → 추가로 `documents(type='final_invoice')` 발행 (별도 slug)
- `quote_groups`/`quote_items`/`quote_group_members` → 각 document별로 복제 (양쪽 문서에)

기존 quotes 테이블은 일단 보존 (Phase 2c에서 DROP 예정).

## 5. Documents 모델 — Phase 2c (방향성 추가)

사용자가 "agent가 client에게 deposit 발행하는데 admin 계좌가 적힌 invoice는 이상함" 짚어줘서 큰 재설계.

### 5개 돈 흐름 매핑
| # | Flow | from → to | 발행 위치 |
|---|---|---|---|
| 1 | Client → Agent (디파짓) | agent → client | **Agent UI** |
| 2 | Agent → Admin (디파짓 forward) | admin → agent | **Admin UI** |
| 3 | Client → Admin (잔금) | admin → client | Admin (Finalize 자동) |
| 4 | Client → Admin (추가) | admin → client | Admin |
| 5 | Admin → Agent (커미션) | **agent** → admin | **Agent UI** (travel 후만) |

### 스키마 추가
```sql
ALTER TABLE documents ADD COLUMN from_party TEXT NOT NULL;  -- admin/agent
ALTER TABLE documents ADD COLUMN to_party TEXT NOT NULL;    -- client/agent/admin
```

기존 행 backfill (commission은 agent→admin으로 flip).

### CaseDocumentsSection actor prop
- `actor='admin' | 'agent'` — 어떤 발행 버튼 보일지 결정
- `canEdit = actor === doc.from_party` — Mark Paid / Add/Remove Item 권한
- agent 사이드: `+ Deposit (to Client)`, `+ Commission (to Admin)` (travel completed 후만)
- admin 사이드: `+ Deposit Settlement (to Agent)`, `+ Additional (to Client)`

### QuoteDocument 분기
- `from_party='agent'` → agent의 `bank_info` JSONB + agent name signer
- `from_party='admin'` → `system_settings.bank_details` + signer_snapshot
- "To" 라벨도 to_party 따라 client/agent/admin 분기

### Bank info schema 통일
Agent bank_info JSONB → admin과 같은 6필드:
- bank_address → **address**
- account_holder → **beneficiary**
- + beneficiary_number 신규 (optional)

Setup wizard + Agent Payouts 폼 모두 6필드로. QuoteDocument의 BankDetails block 양쪽 invoice에 동일 렌더.

## 6. E2E 발견 버그들

사용자가 실제로 돌려보면서 잡힌 것들:

### Schedule
- 첫 업로드인데 "What changed?" 박스 보임 → v2+ 일 때만 표시
- `schedules.quote_id` FK가 quotes 테이블 가리킴 → 신규 케이스 (quotes 비어있음) 업로드 실패. FK constraint DROP

### UI 톤
- Hero 매우 잘 보이는데 정작 본 action 박스(스케줄)는 묻힘 → status 따라 매칭 톤 (reviewing_schedule=violet, awaiting_payment=cyan)
- Confirmed schedule의 Admin Note가 cyan으로 시선 분산 → confirmed 시 muted gray
- Pricing 입력 자동 천단위 콤마

### Documents UI
- "+ Deposit Settlement" → agent의 deposit invoice 없으면 disabled (forward는 deposit 발행 후만)
- "+ Additional" → Balance(=final) invoice 없으면 disabled
- 알림 방향 정정: issuer는 자기에게 알림 X, counterparty/admin에게만
- Deposit (to client) vs Deposit Settlement (to agent) 시각 구분 — 다른 톤 + "Agent → Client" / "Admin → Agent" 라벨

### Critical buf — admin → admin 표시
사용자 "Agent가 client에게 deposit invoice 발행했는데 왜 admin → admin?" 직격.
- DB 확인하니 **올바른 값** (agent/client) 들어있음
- UI는 "Admin → Admin"으로 표시
- 원인: `caseData.documents(...)` 쿼리에서 `from_party, to_party` 컬럼 누락 → 클라이언트 undefined → `partyLabel(undefined)` fall-through로 'Admin'
- fix: admin/cases/[id] + agent/cases/[id] 쿼리에 `from_party, to_party, payment_received_at, created_at` 추가

### Items 안 보이는 버그
- `issueDepositInvoice` 가 group_id 없이 item 추가
- QuoteDocument는 `document_groups → items` 중첩 쿼리만 사용 → orphan items 누락
- Total $0.00 / "No items" 표시
- fix: ungrouped items 별도 fetch + flat 렌더 (deposit/commission/additional처럼 그룹 필요 없는 type용)

## 9. 회고

### 이번 세션이 오래 걸린 이유 (사용자 지적)
1. **작업 종류**: UI 폴리시 → 아키텍처 갈아끼우기. 본질적으로 무거움
2. **같은 영역 재방문**: Documents 만 5번 갈아엎음 (Phase 1 → 2a → 2b → 2b 폴리시 → 2c 방향성). 매 단계 이전 단계 위에서 또 수정
3. **내 실수 cycle**: regex 버그 (`--` → `,`), name dedup 대소문자, NBSP false alarm, IIFE 위치 등 5~15분씩 까먹은 게 합쳐서 1시간+
4. **세션 길이**: 컨텍스트 누적으로 후반부 응답 느려짐
5. **빌드 횟수**: 안전상 자주 돌렸지만 매 30s~2min

### 다음에 빠르게 가는 법
- 큰 변경은 design-first → 데이터 모델 / 흐름 다이어그램 먼저 합의
- 세션 짧게 끊기 (5/1처럼 14시간 단일 세션은 비효율)
- 빌드 마일스톤 단위로만
- 응답 markdown 짧게 (요약 길게 쓰는 게 시간 잡아먹음)
- E2E 빠른 round 자주 끼우기 (push 전 사용자 검증)

특히 **테스트 없이 push 하지 말 것**. 이번 세션 막판에 fetch 쿼리에서 from/to 누락 채로 commit/push해서 사용자 화나게 함. 부끄러움.
