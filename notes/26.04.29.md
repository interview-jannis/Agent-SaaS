# 연구노트 — 2026.04.29

## 작업 개요

Cases 페이지(Admin/Agent) 여백·테이블 시각 이슈 디버깅. 작은 작업이지만 여러 단계 거침.

---

## 1. Cases 풀폭 → 카드형 복원

### 배경
어제 데이터 테이블 전반을 풀폭(no horizontal padding)으로 통일했었음 (Stripe/Linear 패턴). Cases도 거기 따라갔다가 사용자가:

> "음 cases는 여백이 있는 게 더 나은 것 같아. 다른 건 지금 유지하자"

### 결정
- **Cases (Admin/Agent)**: `px-12 py-8 space-y-6` 외곽 padding + 카드형 wrapper 복원
- **다른 데이터 테이블**(Clients/Agents/Audit): 풀폭 유지

### 이유
Cases는 그룹 헤더(상태 색 pill) + 케이스 row가 시각적으로 분리돼야 함. 풀폭이면 그룹 헤더가 viewport 끝까지 뻗으면서 산만해짐. 카드형 wrapper 안에 들어 있어야 그룹 구분이 자연스럽게 시선에 잡힘.

복원한 스타일:
- 외곽: `px-12 py-8 space-y-6`
- Pill 바: `bg-[#0f4c35]/[0.04] border border-[#0f4c35]/15 rounded-xl shadow-sm px-4 py-2.5`
- 테이블 wrapper: `bg-white rounded-2xl border border-gray-100 overflow-hidden`

## 2. Cases 테이블 시각 이슈 디버깅

복원 직후 사용자 캡처:
> "Completed · Awaiting Payout 위에 흰 셀이 있고, #C-004는 그룹 헤더에 딱 붙어 있는데 #C-003은 그 위에 흰 여백이 꽤 있다"

### 1차 분석 + 수정

**(a) 흰 셀 (thead와 첫 그룹 헤더 사이 갭)**
- 원인: 테이블이 `border-collapse: separate` (Tailwind 기본) → thead/tbody 사이 기본 border-spacing
- 수정: `<table>`에 `border-collapse` 클래스 추가

**(b) 세로 정렬 / 행 높이 불일치**
- 원인: Settlement 컬럼이 케이스마다 다른 콘텐츠 (Pending pill vs "—" dash)
- pill은 `px-2 py-0.5 rounded-full border` 추가 padding으로 더 큼
- 수정: dash도 동일 padding + `border-transparent`로 invisible border 적용 → 행 높이 균일화

→ 사용자 재확인 결과 여전히 이슈 남아있음

### 2차 분석 + 수정

테이블 내부 다른 layout 흐트러뜨림 가능성 점검:

**(c) `sticky top-14` 그룹 헤더 td**
- `<td>`에 sticky를 쓰면 일부 브라우저에서 셀 박스 모델이 미묘하게 어긋남 (table 안 sticky의 알려진 quirk)
- 그룹이 9개라 어차피 sticky 효과 미미했음 (스크롤 시 다음 그룹 헤더가 즉시 따라옴)
- 수정: sticky 제거

**(d) `border-y border-gray-200` → `border-b only`**
- thead의 `border-b border-gray-100`과 그룹 헤더 td의 `border-t`가 인접하면서 border-collapse 처리에 변동 가능
- 위쪽 border 제거하고 아래쪽만 → border 충돌 깔끔히 해소

**(e) 배경색 위치 이동: `<td>` → `<tr>`**
- `<tr className="bg-gray-100">`로 옮김. 행 전체에 일관 적용

이걸로 흰 여백 이슈 해결.

## 느낀 점 / 메모

- **테이블 layout 디버깅은 자잘한 변수가 많음**: `border-collapse`, `sticky`, `border-y` 충돌, 셀 콘텐츠 높이 차이 등이 누적되어 "흰 여백"이라는 단일 증상으로 보임. 한 번에 한 변수씩 제거하면서 좁혀가는 게 효율적이었음.
- **풀폭 vs 카드형은 페이지 성격에 따라 다름**: Cases는 그룹핑이 핵심이라 카드형이 맞고, 단순 리스트(Clients/Agents)는 풀폭이 운영적. 모든 데이터 테이블을 같은 패턴으로 통일하려던 게 부정확했음.
- **빠른 자기 교정 가치**: 어제 풀폭으로 통일했다가 오늘 사용자 한마디에 Cases만 분리. UI는 사용자가 실제로 보면서 결정하는 게 정답이라 첫 시도에 정착하려 욕심내지 않는 게 나음.
