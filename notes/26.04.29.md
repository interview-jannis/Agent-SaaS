## 1. Status 8단계 재설계 (사용자 작업, 코드 lib에 반영)

### 새 lib `src/lib/caseStatus.ts`

```ts
export type CaseStatus =
  | 'awaiting_info'         // (NEW) 정보 등록 단계
  | 'awaiting_schedule'     // 견적 발송 후, 스케줄 작업 대기
  | 'reviewing_schedule'    // 스케줄 업로드, agent 검토 대기
  | 'awaiting_pricing'      // (NEW) agent 컨펌, admin 가격 finalize 대기
  | 'awaiting_payment'      // 인보이스 발송, 결제 대기
  | 'awaiting_travel'       // 결제 확정, 여행 대기
  | 'completed'             // (RENAMED from travel_completed)
  | 'canceled'              // (NEW) 취소 종결
```

### 변화 핵심
- **`awaiting_info` 신규**: quote_sent 전 단계. 정보 등록·기본 검증
- **`awaiting_pricing` 신규**: 흐름 B의 finalize 단계가 명시적 status로 분리됨
- **명명 통일**: 모든 상태가 `awaiting_*` / `completed` / `canceled` — forward-looking, 라벨과 1:1
- **`canceled` 추가**: 이전엔 케이스 자체 삭제했지만 이제 종결 상태로 보존 (audit / 분석 가능)

### 파생 상수
- `ALL_STATUSES`, `PIPELINE_ORDER`, `STATUS_LABELS`, `STATUS_STYLES`
- `STATUS_OWNER`: agent / admin / customer / none — Action Required 큐 분기 기준
- `CANCELLABLE_STATUSES`: 결제 전 5개 상태 (cancel 가능)
- `ACTIVE_STATUSES`: 6개 (terminal 제외, "in pipeline")

### `src/lib/caseTransitions.ts`
status 전환 로직을 lib으로 추출 (그동안 페이지에 흩어져 있던 것).

## 2. Cases 페이지 그룹핑 UX (Admin + Agent)

### 동기
> "데이터 적은 지금은 별 문제 없는데, 나중가면 되게 복잡해질 것 같은 느낌"

100~500건에서 case#만 보고 찾기 어려움 → 그룹핑 + 검색.

### 구조 (Admin > Products 패턴 차용)
```
┌── 헤더 (h-14) ────────────────────────┐
│ Cases [N]    검색 입력 ───── 우측      │
├──────────────────────────────────────┤
│ JUMP TO  [Pending][Travel][...] 칩 바 │  sticky top-0
├──────────────────────────────────────┤
│  thead: Case # | Lead | Travel | ... │
│  ─── Completed · Pending Settlement ──│
│  #C-004                              │
│  ─── Awaiting Travel (0) ────────────│
│   No cases                           │
│  ─── Awaiting Payment ───────────────│
│  #C-003                              │
│  ...                                 │
└──────────────────────────────────────┘
```

### Display order — **Pending Settlement 최우선** (사용자 결정)
```ts
const DISPLAY_ORDER = [
  'completed_pending',     // ⭐ 돈 오고가는 게 가장 중요
  'awaiting_travel',       // 결제 끝, 여행 임박
  'awaiting_payment',
  'awaiting_pricing',
  'reviewing_schedule',
  'awaiting_schedule',
  'awaiting_info',
  'completed_settled',     // 종결 (gray)
  'canceled',              // 종결 (rose)
]
```

`completed`는 가상으로 둘로 분리:
- **completed_pending** (amber, 최상단): 정산 미완 — admin이 챙겨야 할 것
- **completed_settled** (gray, 하단): 완전 종결 — 들여다볼 필요 없음

분리 기준:
- Admin: 파트너 송금 + 에이전트 정산 둘 다 완료 시에만 `settled`
- Agent: 자신의 정산 받았는지만 (Received vs Awaiting Payout)

### 그룹 안 정렬: 여행 임박순
```ts
function sortByTravelImminent(a, b) {
  // travel_start_date 오름차순, null은 뒤로
}
```

### Empty 상태도 항상 표시
- 칩: 회색 disabled `(0)`
- 테이블: 그룹 헤더 + "No cases" italic 작은 텍스트
- 이유: 칩과 테이블이 같은 정보 — "이 단계에 0건"이 즉시 인지됨

### 검색
- Admin: case# / 에이전트 이름 / 클라이언트 이름
- Agent: case# / 클라이언트 이름

## 3. UI 일관성 대청소

### 사용자 지적
> "어떤 건 제목 글씨 크기가 크고 또 어떤 건 작아"
> "어떤 건 표 좌우 여백이 있는데, 어떤 건 없어. 그 여백 크기도 다 다름"
> "검색창 위치도 탭 별로 다 다르고"

### 표준 정립

**(A) 헤더 패턴 통일** — Pattern B 단일화 (h-14 top bar)
- 4개 예외(Overview, Products, Categories, Settings)도 모두 Pattern B로 전환
- 큰 inline 타이틀 → small top bar 타이틀

**(B) 타이틀 크기**
- `text-sm` → **`text-base`** (한 단계 키움)
- 전체 16개 페이지 일괄 적용

**(C) 콘텐츠 padding 2-tier**
| 타입 | padding | 페이지 |
|---|---|---|
| 대시보드 | `px-12 py-8 space-y-6` | Overview, Dashboard, Payouts, Settlement, Audit |
| **데이터 테이블** | **full-width** (no horizontal padding) | Clients, Agents 등 |
| **Cases (예외)** | `px-12 py-8 space-y-6` | Cases (사용자가 별도로 여백 선호) |
| 폼/설정 | `max-w-* mx-auto px-6 py-8 space-y-6` | Settings, Categories, Profile, Contracts |

**(D) 검색·필터·액션 위치**
- 항상 `ml-auto flex items-center gap-2` 그룹으로 우측 정렬
- Admin Agents의 회색 박스형 검색 → 일반 input 통일

### 근거
- 데이터 테이블은 Stripe/Linear/GitHub 패턴 차용 — full-width로 정보 밀도 우선
- 대시보드(카드 위주)는 양쪽 여백 — 카드 정렬·breathing room
- 폼은 max-w + 중앙정렬 — 가독성 우선

### Cases는 예외
처음엔 풀폭으로 갔다가 사용자가 "여백 있는 게 더 나음" → 다시 카드형 복원. 그룹 헤더 + 케이스 row가 시각적으로 구분되어야 함, 풀폭은 산만했음.

## 4. Vercel 빌드 에러

### 증상
```
./src/app/admin/agents/[id]/page.tsx:599:105
Type error: Element implicitly has an 'any' type because 
expression of type 'string' can't be used to index 
type 'Record<CaseStatus, string>'
```

### 원인
`CaseRow.status: string` 였는데 `STATUS_STYLES[c.status]`는 `Record<CaseStatus, string>` 인덱싱. 로컬 tsc는 통과(incremental cache)했지만 Vercel strict 체크는 잡음.

### 수정
`CaseRow.status` → `CaseStatus` 타입으로 강타입화.

→ Push → Vercel 재빌드 통과 ✓

## DB 변경

오늘 새 마이그레이션 없음. 기존 status 값들은 새 명명 규칙으로 마이그레이션 (사용자 작업):
- `quote_sent` → `awaiting_schedule`
- `schedule_reviewed` → `reviewing_schedule`
- `schedule_confirmed` → 분리 (`awaiting_pricing` + `awaiting_payment`)
- `payment_completed` → `awaiting_travel`
- `travel_completed` → `completed`
- 신규: `awaiting_info`, `canceled`

## 느낀 점 / 메모

- **Status 8단계는 사용자가 흐름 정리 의도로 설계한 것**. `awaiting_info`와 `awaiting_pricing`이 추가되면서 case lifecycle이 더 정교해짐. 코드는 lib 하나에 모이니까 변경에 강함. 이전엔 7~8개 파일에 STATUS_LABELS 흩뿌려져 있었음 — 이제 한 곳.
- **그룹핑 UX의 핵심은 "Pending Settlement 최상단"**: 단순 status 순서가 아니라 "내가 챙겨야 할 우선순위"로 재정렬. 사용자 직관과 정확히 매칭.
- **Empty 그룹 표시**: 처음엔 hidden이었다가 사용자가 "0인 것도 보였으면" — 한 번에 모든 단계 인지하는 게 더 운영적. 의외로 정보 밀도 높지 않음.
- **풀폭 vs 카드형 padding**: 테이블 종류별로 답이 다름. Cases는 그룹 헤더와 case row가 시각적으로 분리 필요해서 카드형이 맞음. Clients/Agents는 단순 리스트라 풀폭이 더 운영적.
- **Vercel strict TS와 로컬 tsc 차이**: incremental cache 때문에 로컬 통과해도 Vercel에서 잡히는 케이스 종종 있음. CI에서만 잡히는 패턴 — 향후 commit 전 `tsc --noEmit` 한 번 더 도는 게 안전.
- **연구노트 작성 자체의 가치**: 오늘 작업 중간중간 사용자 결정/번복이 많았는데(테이블 풀폭 → 카드 복원, sticky 적용 → 제거 등), 노트로 정리하니까 "왜 이렇게 됐나" 컨텍스트가 보존됨. 한참 후 보면 답답할 결정도 당시 이유가 있음.
