## 1. Cases 페이지 폴리시 (오전)

### 풀폭 → 카드형 복원
어제 데이터 테이블 전반을 풀폭(Stripe/Linear 패턴)으로 통일했었음. Cases도 거기 따라갔다가 사용자 피드백으로 카드형 복원.

- **Cases (Admin/Agent)**: `px-12 py-8 space-y-6` 외곽 padding + 카드형 wrapper 복원
- **다른 데이터 테이블**(Clients/Agents/Audit): 풀폭 유지

이유: Cases는 그룹 헤더(상태) + 케이스 row가 시각적으로 분리돼야 함. 풀폭이면 그룹 헤더가 viewport 끝까지 뻗으면서 산만.

### 테이블 시각 이슈 디버깅
복원 직후 "Completed · Awaiting Payout 위에 흰 셀, #C-004는 그룹 헤더에 딱 붙어 있는데 #C-003은 그 위에 흰 여백" 이슈.

원인 누적:
- `border-collapse: separate` (Tailwind 기본) → thead/tbody 사이 갭
- `<td>`에 sticky → 셀 박스 모델 미묘한 어긋남 (table 안 sticky의 알려진 quirk)
- `border-y` 인접 충돌

수정:
- `<table>`에 `border-collapse` 추가
- 그룹 헤더 td의 sticky 제거 (효용 미미했음)
- `border-y` → `border-b only`로 충돌 해소
- 배경색을 `<td>`에서 `<tr>`로 이동

## 3. Cases 리스트 톤 통일 라운드

### 발단
"Cases 4건뿐인데 page에 status 9개가 모두 펼쳐져서 텅텅 빈 느낌." 처음엔 빈 섹션 숨김 제안 → 사용자가 "순서/흐름은 보여야 함"이라 원복 요청.

### 절충안 (사용자 의견 수렴)
**섹션 헤더 pill**은 유지(파이프라인 흐름 보존) + **빈 body의 "No cases" 행만 제거**(공간 절약). 9개 섹션 중 6개 빈 거 × ~50px = ~300px 회수.

### 색 노이즈 제거
9개 status pill이 다 다른 색이라 무지개. 위→아래 위치로 순서는 이미 전달됨, 색은 노이즈. **monochrome 통일**:
- 활성 그룹: `text-gray-800 font-semibold` (또렷)
- 빈 그룹: `text-gray-400` (있는 듯 없는 듯)
- pill 모양 제거 → uppercase 라벨 + tracking-wide

**메모리 저장**: "Cases 리스트 빈 status 섹션 숨기지 말 것" — 이전에 한 번 시도했다가 사용자가 되돌린 결정이라 다음에 다시 제안 안 하도록 feedback 메모.

### Settlement 컬럼 정리
`● P · ● A` 점만 있던 게 암호 같다는 피드백.

진화 과정:
1. `[✓ Partner] [⋯ Agent]` chip + 색 → 사용자 "지저분해 보임"
2. **chip 껍질 제거, 마커만 색 살림**: `✓ Partner ⋯ Agent` 평문 + 마커 한 글자에만 (✓ 녹색 / ⋯ 앰버)
3. Agent쪽도 동일하게 (Received / Pending)

### JUMP TO 헤더 통합
본문 sticky 박스로 ~60px 차지하던 JUMP TO를 헤더 바 안으로 이동. 사용자 제안: "헤더는 어차피 고정 영역이니까 거기 둬".

- bg-gray-50 + 좌측 "JUMP" 라벨 (세로 divider 구분) + chip들
- chip 라벨 다이어트: "Awaiting" 5번 / "Completed ·" 2번 반복 제거
  - `Pending Payout · Travel · Payment · Pricing · Schedule Review · Schedule Prep · Client Info · Settled · Canceled`
- chip hover에 풀 라벨 tooltip
- 좁아지면 가로 스크롤 (`overflow-x-auto no-scrollbar`)

### 자잘한 거
- `Case #` → `Case` (헤더 4곳)
- `~` → en-dash `–` (Travel period 5곳)
- Members 컬럼 `text-center` → 좌측 정렬
- Lead Client에 `#CL-001` 추가 (Agent 컬럼이랑 동일 패턴)
- Admin layout `overflow-y-auto` → `overflow-hidden` (Agent와 통일, outer 스크롤바로 인한 width 차이 해소)

### 섹션 헤더에 의미 아이콘
사용자 제안 — dot보다 의미있는 작은 SVG가 낫다.

각 status별 monochrome 아이콘:
- Client Info → user
- Schedule Prep → upload arrow
- Schedule Review → eye
- Pricing → tag
- Payment → receipt
- Travel → paper plane
- Pending Payout → check-badge
- Settled → check-circle
- Canceled → x-circle

## 5. Invoice Signer (PIC) 시스템

### 배경
사용자 질문: "Quotation/Invoice 발신자에 사람 이름 넣는 게 나을지". 4/24 미팅 때 결정한 PIC 추가 여부 결정 사안.

### 결정
- 회사 내부 시뮬에서 직원이 admin 역할 돌아가면서 한다 → admin 여러 명
- 따라서 옵션 B (admin profile 기반) 채택
- Quotation은 finalize 전이라 PIC 없음 (회사명만)
- Invoice는 finalize한 admin 이름 + 직책 표시

### 구현
**DB**:
```sql
ALTER TABLE admins ADD COLUMN title TEXT;
ALTER TABLE quotes ADD COLUMN signer_snapshot JSONB;
```

**Settings 새 row "My Display Info"**:
- per-admin (auth_user_id 매칭)
- Name + Title 입력 (예: "이지수" / "Account Manager")
- Inline edit 패턴

**Pricing Finalize 로직**:
- 매 save마다 현재 admin의 name+title을 `quotes.signer_snapshot`에 갱신
  - 처음엔 "첫 finalize에만 동결"이었는데 기존 invoice 갱신 안 되는 문제 → 매 save로 변경
  - 의미: signer는 "마지막으로 invoice 책임진 admin"

**QuoteDocument 표시 (Invoice 모드만)**:
- 헤더 "From" 영역: `: 이지수 (Account Manager)` + 회사명 한 줄 아래
- 하단 서명: Yours Faithfully → 이름 → 직책 → 회사명
- signer_snapshot 없으면 (구 invoice) 회사명만 노출 — 하위호환

## 6. Super Admin 시스템

### 동기
사용자: "직원 몇 명이 될지 모르겠는데 매번 Supabase에서 admin 등록하는 건 귀찮음. 최고관리자 한 명 지정해서 그 사람이 새 admin 만들 수 있게."

### 권한 정책 (사용자 요구 반영)

| 영역 | 모든 admin | Super admin only |
|---|---|---|
| Cases / Agents / Clients / Products / Settlement / Audit / Overview | ✓ | (동일) |
| Settings: My Display Info / Change Password | 조회+수정 (본인) | (동일) |
| Settings: Exchange Rate / Company Margin / Bank Account | **조회** | 수정 |
| Contracts (Orientation/NDA/Partnership) | **조회** | 수정/업로드/삭제 |
| `/admin/admins` (관리 페이지) | 안 보임 | 전체 관리 |

핵심: **조회는 다 가능, 수정만 super admin**. 일반 admin도 환율/마진/은행 운영 컨텍스트 알아야 함.

### 구현

**DB**:
```sql
ALTER TABLE admins ADD COLUMN is_super_admin BOOLEAN DEFAULT false;
UPDATE admins SET is_super_admin = true WHERE email = 'xxx';
```

**`/admin/admins` 페이지 신규**:
- Super admin만 접근 (regular admin 가면 "Super admin access required")
- 카드 리스트: 사진/이름/이메일/직책/Super Admin 뱃지/You 표시
- + Add Admin 버튼 → 모달 (Name / Email / Title / Temp Password)
- 임시 비번 12자 안전 자동 생성, 손수 변경 가능, ↻ Regen 버튼
- 생성 완료 시 Email + Password가 한 번 노출 + Copy 버튼 (Slack로 직접 전달)
- "이 화면 닫으면 비번 다시 못 봄" 경고
- 본인은 삭제 불가 / 다른 super admin도 삭제 불가 (안전장치)

**API**:
- `POST /api/admin/create-admin` (service role + super admin 검증)
- `POST /api/admin/delete-admin` (자기 자신/다른 super admin 삭제 방지)

**Sidebar**:
- "Admins" 메뉴는 super admin만 보임
- 하단 라벨: super는 "Super Administrator", 일반은 "Administrator"

**이메일 발송 vs 매뉴얼**: 매뉴얼 채택. Resend가 아직 미통합 + 같은 회사라 슬랙으로 직접 전달이 더 빠름. agent invite 패턴이랑 일관.

### 후속: Admin 생성 → 초대 링크 방식으로 리팩토링

처음엔 super admin이 직접 모든 정보(name/email/title/temp password)를 입력하는 방식으로 만들었는데, 사용자 의견:
> "agent 처럼 링크만 생성되는 게 좋지 않을까? 계약 같은 거 없이, 링크 받으면 바로 이메일 등록하고 비밀번호 등록하는 쪽으로"

타당함. 이유:
- 보안: 임시 비번이 슬랙에 노출되지 않음
- 단순: super admin은 버튼만 누르면 됨 (4개 필드 입력 X)
- 셀프 서비스: 신규 admin이 자기 정보를 직접 입력 (심리적 주체성)
- agent invite 패턴이랑 코드도 통일됨

**구현**:

DB 컬럼 추가 (agent invite와 동일 패턴):
```sql
ALTER TABLE admins ADD COLUMN invite_token TEXT UNIQUE;
ALTER TABLE admins ADD COLUMN invite_secret TEXT;
ALTER TABLE admins ADD COLUMN invited_at TIMESTAMPTZ;
ALTER TABLE admins ADD COLUMN invite_expires_at TIMESTAMPTZ;
```

API 라우트 3개:
- `/api/admin/invite-admin` — super admin이 호출, placeholder auth user + admins row + token 생성, invite URL 반환
- `/api/admin/claim-admin-invite` — 토큰 검증 + placeholder credential 반환 (signInWithPassword용)
- `/api/admin/setup-admin` — Name/Email/Title/Password로 admin row + auth user 갱신, 토큰 무효화

페이지:
- `/admin-invite/[token]` — claim → 자동 로그인 → setup 폼 (Name/Email/Title/Password/Confirm) → submit → /admin/overview

`/admin/admins` 페이지 변경:
- 버튼 라벨: "Add Admin" → "Invite Admin"
- 모달 입력 필드 0개 (Generate Invite Link 버튼만)
- 결과로 invite URL + 만료일 + Copy 버튼 노출
- pending invite 상태도 admin 카드로 노출 ("Invited admin · Awaiting setup · expires...")
- pending인 경우 Delete 라벨이 "Cancel"로 변경

만료: 7일 (`invited_at + 7d`). claim 시 서버측 만료 검증 + 사용 후 즉시 토큰 무효화 (단일 사용).

구 `/api/admin/create-admin` 라우트 제거.

| Country 입력 | `<datalist>` 자동완성 | 오타 방지 + 자유 입력 허용 |
| Invoice signer | 매 save마다 갱신, snapshot 동결 | 기존 invoice도 업데이트 가능 |
| 권한 정책 | 조회는 모두, 수정은 super only | 운영 컨텍스트 필요, 책임만 분리 |
| Admin 신규 등록 | 매뉴얼 비번 전달 (Resend X) | MVP 단순, Slack 즉시성 |
| Admin 등록 흐름 | 직접 생성 → **초대 링크**로 리팩토링 | 비번 노출 X, super admin 입력 0개, 셀프 서비스 |

## 느낀 점 / 메모

**iteration의 가치**:
- Settings 레이아웃은 4번 시도 끝에 정착. 매번 사용자가 작은 신호를 줬고("어색하다", "save가 중간에 떠있다") 조금씩 각도를 바꾼 끝에 GitHub-style inline edit이라는 정답에 도달. 첫 시도에 정착하려 하지 말고 작은 피드백을 따라가는 게 효율적.

**색 비용의 인지**:
- 모노크롬으로 가는 결정이 반복됨 (status pill, Hero 4지표, 차트, dot, Settlement chip). 색은 의미를 전달하지만 비용도 큼 — 한 화면에 색이 많으면 정보 위계 무너짐. "한 가지 색은 한 가지 의미만 책임"이 원칙.

**권한 분리의 설계**:
- 처음엔 super admin만 보이게 하려 했는데 사용자가 "조회는 필요"라고 정정. 권한 분리는 binary가 아니라 "view" / "edit" / "manage"로 세분화해야 함. 이건 운영하면서 가다듬을 가치 있음.

**소규모 팀 SaaS의 현실**:
- "이메일 자동 발송 vs 매뉴얼 복사" — 회사 내부면 Slack이 이메일보다 빠름. SaaS의 "이상적 자동화"가 항상 운영적 정답이 아니라는 걸 다시 확인. MVP에선 매뉴얼이 우월할 때가 많음.
